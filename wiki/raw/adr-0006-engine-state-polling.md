# Surface engine-driven state changes by gated polling, not the WebSocket

ADR-0005 shipped Podium as three independent processes — the Symphony scheduler under `symphony-host.service`, the FastAPI/uvicorn API under `podium-api.service`, and the Next.js frontend under `podium-web.service` — and gave the API a WebSocket (`WS /api/ws`, #017) that pushes Run state transitions and Issue field changes so the board stays live without HTTP polling. In practice the WebSocket only carries mutations that originate **inside the API process**: the `WebSocketHub` is an in-process fanout, and the only `run.updated`/`issue.updated` publishes happen on API-handled writes (operator PATCH/create) and at API startup for seeded rows (`web/api/main.py:126`). The scheduler is a separate process that writes Run and Issue rows **directly to the shared SQLite store** via `tracker_podium`; it never touches the API's hub. The consequence, surprising to anyone who reads "#017 added live updates" and assumes the board reflects the engine in real time: the exact state changes an operator watches for — an Issue going `todo`→`running`→`in_review`, a Run finalizing `succeeded`/`failed`, a binding falling quiet — are produced by the scheduler and are therefore never pushed. The board and run-detail views only catch up on a manual refetch or a TanStack tab-refocus. The run log compounds this: `agent_runner.py:272` uses `process.communicate(timeout=…)`, which blocks until pi exits and captures all stdout/stderr at once, so a Run's log lands only at completion — there is no incremental output to stream even if the transport existed.

We chose to surface engine-driven state changes with **gated frontend interval polling** rather than bridge the scheduler into the WebSocket. TanStack Query's `refetchInterval` runs fast (~3s) on the board and any open run-detail panel only while an Issue is `queued` or `running`, and slows or stops when the board is idle, keeping steady-state cost near zero against a local WAL-mode SQLite read (#019). The WebSocket is retained for what it actually does well — instant optimistic feedback on the operator's own UI actions (create, PATCH). Run liveness is built on the same lever: an elapsed timer ticks client-side from `started_at` for in-flight Runs, and the detail panel's metadata, duration, and log refresh through the poll the moment pi exits — no live log tail, because the dispatch path does not produce one.

The alternatives were rejected for concrete reasons. **Scheduler→API notify** (the scheduler POSTs a localhost API endpoint after each mutation, which then fans out over the existing WebSocket) gives true sub-second push but makes the scheduler depend on the auth-gated API process being up, requires an internal authentication path through the #018 gate, and adds failure handling for when the API is down — real coupling between two units ADR-0005 deliberately made fail independently, bought for a latency improvement a single operator watching a board does not need. **An API-side DB poller** (the API polls SQLite for changed rows by `updated_at`/`last_event_at` and publishes over the WebSocket) centralizes polling so one server-side loop feeds every browser, but that advantage only matters with multiple concurrent clients, and this is a single-user system; it adds a background task and change-detection bookkeeping to the API for no real gain here. **True live log streaming** (rewrite dispatch to read pi stdout line-by-line, stream it to the log file, and push lines over the WebSocket) was scoped out: it touches the core dispatch path and entangles the silent-exit guard (`agent_runner.py:281`) that inspects full stdout/stderr, and runs are already bounded by `run_timeout_ms` with orphans reaped by the #022 restart reconciler, so the "is it stuck" question polling cannot answer is already answered elsewhere.

The accepted costs: surfacing is pull-not-push with ~3s latency while active, and each browser tab polls independently — both immaterial for a single-operator homelab console. The decision is reversible: if multi-client or sub-second push ever matters, the scheduler→API notify path can be added against the same WebSocket the optimistic-UI path already uses, at which point polling can be narrowed or dropped. This ADR records *why* a system that already has a WebSocket nonetheless polls for the data that matters most, so the next reader does not "fix" the polling by assuming the WebSocket already covers engine state.
