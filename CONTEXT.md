# Symphony

Symphony is the scheduler that polls a Plane project for candidate issues, dispatches each to a coding agent against a bound git repo, and posts the agent's results back to the issue. Originally built to manage the homelab infrastructure repo; being generalized to serve any repo.

## Language

**Symphony**:
The engine: poll Plane → select candidate issue → render prompt → dispatch to an agent → comment results back. One engine, many projects.
_Avoid_: "the bot", "the runner"

**Podium**:
The operator console and native tracker that replaces Plane as Symphony's human-facing surface. Symphony owns Podium's schema, lifecycle, AI-shaped metadata, and live update stream end-to-end; the [[tracker-adapter]] seam stays, with Podium as one more implementation alongside Plane. Plane is retained only as a transitional safety net during binding-by-binding migration; once both bindings cut over, Plane is archived. Podium is paired with Symphony the way a podium is paired with a conductor: the surface the operator stands at, distinct from the engine that does the conducting.
_Avoid_: "the UI" (Podium is the tracker too, not just a frontend), "Symphony Web" (reads as a sub-module rather than a thing), "the dashboard" (Podium is the primary control surface, not a debug view)

**Project Binding**:
The mapping that ties one tracker project to one git repo plus its dispatch config. The unit Symphony iterates over. A binding carries: its tracker kind (`podium` today; Plane retained only for rollback/dormant adapter compatibility), the repo path and base branch; `default_agent` (`pi` or `claude`, see [[Agent]]); the per-binding **agent-reaction capabilities** (see below); an `approval.enabled` flag (off for the current bindings); and the [[Landing]] policy. **Coding bindings** (today: `trading` and `symphony`) skip Symphony's schedule, blocked reconciler, and approval gate — the agent owns its own git operations and Symphony performs no landing step. **Infra bindings** (today: `homelab`) run all three and may project approval/schedule roles onto Podium issue columns.

**Agent-reaction capabilities (ADR-0032, `proposed`):** the `infra`/`coding` **binding_type** was a *bundle* — one flag (`is_coding`) gated six unrelated behaviours (the [[Workflow]] preamble, verify-the-cure/block-vs-schedule policy, maintenance-window scheduling, blocked-reconciler cure, the approval gate, and auto-close-on-verified). ADR-0032 decomposes the bundle: the *policy/prompt* behaviours move into a **project-supplied preamble** (the project defines how its agents react), the *engine capabilities* become orthogonal per-binding opt-in flags (scheduling and blocked-reconciliation joining the already-independent `approval.enabled` and `auto_close_on_verified`), and **binding_type degrades to a thin preset** over those flags — no engine branch reads it directly. The goal is an environment-agnostic engine: the same Symphony serves homelab and a separate production environment by supplying a different preamble + capability set, with no engine code change per environment. Until ADR-0032 lands, `binding_type` still gates behaviour directly in code.
_Avoid_: "workspace" (that is a Plane-era concept above the project), "binding_type gates behaviour" (post-ADR-0032 it is an inert preset; capabilities are the engine-facing surface)

**Remote Binding**:
A Project Binding whose repo checkout lives on another trusted LAN/tailnet host and whose binding entry carries a `remote:` transport target. It remains a normal coding binding from the operator's perspective: issue prompt, preferred Skill, pi RPC session continuity, and Steering should match local coding bindings. The difference is location, not workflow — Symphony stays central while the agent process runs in the remote checkout over SSH. **Multiple remote bindings may share one host** (same `remote.host`+`remote.user`, different `repo_path` and distinct binding `name`) with no collision — SSH dispatch, worktrees, and skill-sync are all per-binding/per-real-host. When they do, they group under one sidebar header via [[Host Alias]] (ADR-0039).
_Avoid_: "remote agent" or "symphony-remote" (there is no deployed Symphony component on the remote host), "special binding" (the target is local coding binding parity)

**Host Alias**:
The optional `remote.host_alias` field on a [[Remote Binding]] — a **display-only** friendly label that determines which sidebar group the binding falls under, decoupled from the SSH transport target (`remote.host`). It exists because a host reached by raw IP (e.g. the n8n Tailscale IP, since bare `n8n` hits a NetBird address where sshd refuses) would otherwise fall back to each binding's own `display_name` as the group key, splitting sibling bindings on one host into separate headers. Stored lowercase (uppercased by sidebar CSS), it feeds only the frontend `host` group key — never SSH dispatch, skill-sync host grouping, worktree paths, or the Podium DB (ADR-0039). The onboard/scaffold skill auto-detects a shared host and backfills the same alias onto every binding on it.
_Avoid_: "display host" (implies an alternate SSH target; the alias never connects), "host name" (the SSH host is `remote.host`; the alias is purely a display grouping label)

**Mode**:
Historical: superseded by [[Skill]] in 2026-06-Podium, then drained of behaviour entirely by ADR-0031 (2026-06-30). Plane-era Mode values (`plan`, `build`, `execute`, `conversation`) survive only as an inert compatibility bridge: `_resolve_mode`/`mode_for_skill` still project them for logging and label display, but **no engine branch acts on `plan`/`build` anymore** — plan-vs-build is an instruction the operator writes in the issue body, not an engine-owned mode. Removal of the dormant vocabulary is a deferred cleanup.
_Avoid_: "task type", "stage"

**Agent**:
The coding tool Symphony shells out to in order to do an issue's work. **Two are live: pi and claude**, selected per issue by the [[agent-adapter]] seam (the routing adapter resolves the issue's agent on every dispatch). pi dispatches via `--mode rpc --provider <p> --model <m>` from the bound repo's checkout directory, with the one-shot `--print --no-session` path retained only as a rollback/legacy mode. claude dispatches through an interactive tmux send-keys session (`--permission-mode bypassPermissions --model <m>`, empty provider, bare model id), reintroduced 2026-06-13 after having been dropped at the thin-engine cutover. The seam still lets a third agent be added without touching the engine.
_Avoid_: "model" (the model is a parameter of an agent, not the agent itself), "pi only" (claude is live again as of 2026-06-13)

**Workflow**:
The **portable Symphony harness contract** — the host-agnostic preamble that tells the agent how to operate under Symphony's orchestration (git ownership, input handling, completion/output contract). Its ownership has moved twice. ADR-0016 (landed 2026-06-20) retired the per-repo `WORKFLOW.md` and shipped the contract as the engine-owned `INFRA_PREAMBLE` renderer constant in `prompt_renderer.py`, pushing only safety/autonomy into the host `CLAUDE.md`. **ADR-0032 (`proposed`) reverses the ownership direction**: the engine stops shipping `INFRA_PREAMBLE`, and each [[project-binding]] instead points at a **project-supplied preamble** file in its own repo (absent/empty → the engine renders *no* preamble, pure "issue is the prompt"). This completes ADR-0016's own stated portability goal ("install on a new host → author a `CLAUDE.md`, touch nothing else"), which the hardcoded constant broke by baking infra prose into the renderer. Anything environment-specific — safety, autonomy latitude, how patrols are handled — lives in the project, never in the engine, which stays strict to its dispatch/verdict job.
_Avoid_: "domain overlay" (the homelab-era label-selected fragments; dropped), "autonomy policy" (autonomy is host `CLAUDE.md`), "safety policy" (safety lives in the repo's `CLAUDE.md`/`AGENTS.md`), "per-repo WORKFLOW.md" (ADR-0016 replaced it with a constant; ADR-0032 replaces the constant with a project-supplied preamble), "engine-owned preamble" (post-ADR-0032 the preamble is project-owned)

**Tracker Adapter**:
The seam that isolates tracker-specific reads and writes behind one interface, so the engine talks to "a tracker" rather than to a concrete backend. Borrowed from sortie's vocabulary. Plane is retired for active bindings as of the Podium cutover: all active bindings (`homelab`, `trading`, `symphony`) use Podium, while the Plane adapter remains dormant as the ADR-0002 hedge and rollback aid. The `trading` Plane project was archived 2026-06-11 (#023d) and its rollback contract removed; the `homelab` Plane project and rollback contract are retained (archive deferred to a follow-up issue).
_Avoid_: "Plane client" (the adapter is the abstraction; the Plane client is one implementation of it)

**Tracker Contract**:
The per-binding mapping from the **Roles** Symphony's engine cares about to a specific project's concrete tracker vocabulary (label/state names plus their per-project UUIDs). A **Role** is a thing the engine branches on — the `agent:*` dispatch override, the `approval-required` gate, `approved`, `scheduled`, and the five states (Todo / In Review / Running / Blocked / Done). (`mode:plan` / `mode:build` are still defined as Roles for compatibility but the engine no longer branches on them — ADR-0031.) The engine references Roles, never raw label strings; each [[project-binding]] supplies the names and UUIDs that satisfy them, and a Role a binding omits simply disables that behaviour. This is the concrete shape the [[tracker-adapter]] resolves. Replaces the old single global enum that lived inside the homelab repo. [[Podium]] additionally carries an **Archived** issue state — a tracker-side disposal state that is deliberately *not* an engine Role: the engine never selects archived work (scheduler and reconcilers cannot reach it), and post-run it honors archived as terminal — a Run that finishes on an already-archived issue writes no verdict state transition and its [[Run Worktree]] is torn down, output discarded.
_Avoid_: "the labels" (a Role is the engine-facing abstraction; labels/states are one project's concrete fillers), "domain labels" (patrol/security/infra… were prompt-routing, now [[Workflow]] content, never engine Roles)

**Agent Adapter**:
The seam that isolates each agent's dispatch shape behind a common interface — pi (one-shot subprocess) and claude (tmux send-keys session) are two implementations. The engine selects and drives an adapter without knowing the agent's mechanics.
_Avoid_: "runner" (the engine is the runner; an adapter is one agent's dispatch implementation)

**Done Marker**:
A unique per-run file the agent creates when it finishes, letting Symphony detect completion of a tmux session that has no process exit code. (Originally a printed nonce string scraped from the pane; amended to a file-based contract — see ADR-0001 amendment.)
_Avoid_: "sentinel" (use Done Marker consistently)

**Verdict**:
The outcome of a single dispatch, declared by the agent via a `SYMPHONY_RESULT:` line whose value is one of **done** / **review** / **blocked** (mapping to the Plane states Done / In Review / Blocked). Last occurrence wins; unknown/absent falls through to a heuristic. The agent may also emit a `SYMPHONY_SUMMARY:` line for the human-readable Plane comment. For claude (tmux) the same lines are read from the per-run result file written before the Done Marker; there is no side-effect inspection backstop (parity with pi).
_Avoid_: "ok/failed" (the real vocabulary is done/review/blocked), "exit status" (that is one input, not the verdict itself)

**Run**:
A single dispatch of one issue to one agent — the unit a Verdict describes. By default the agent runs directly in the bound repo's checkout and owns its own git operations; Symphony only renders the prompt, runs the agent, scrapes the [[Verdict]], and transitions the issue. Independently of [[project-binding]] type, an Issue with `worktree_active=true` executes the Run in a [[Run Worktree]] and Symphony [[Landing|lands]] it on Done; the agent still owns the commit. A Run may execute as one autonomous pass, or later as a staged pipeline (research → … → commit) where stages hand off one at a time inside the same Run.
_Avoid_: "job" (a Run is one dispatch; the scheduler tick that may start several Runs is not itself a Run)

**Run Worktree**:
An opt-in Podium per-Issue persistent worktree controlled by the Issue's `worktree_active` column, available to any [[project-binding]] regardless of type (coding or infra). When active, Symphony dispatches in `worktrees/<binding>/<issue_id>` on branch `podium/<binding>/<issue_id>` and leaves the worktree intact on blocked/abort paths for operator inspection. When inactive, the agent runs directly in the bound repo checkout. Remote bindings (ADR-0012) force `worktree_active` off — the agent commits directly in the remote checkout over SSH.
_Avoid_: "workspace" (a tracker-level concept), "checkout" (the shared repo checkout is the non-worktree execution path)

**Landing**:
Converging a completed Run's work back into the repo's base branch. Without an active worktree Symphony performs no landing step — the agent commits (and pushes, if its Workflow allows) directly inside the bound repo's checkout, and the human inspects / merges from there. For **any binding (coding or infra) with `worktree_active=true`**, Podium attempts a fast-forward merge of the worktree branch into the Issue's base branch when the Issue moves to Done, then tears down the worktree. If the base checkout is dirty or the FF-only merge fails (base diverged / conflict), the Issue moves to Blocked and the worktree remains for operator inspection. If the worktree holds work the agent never committed, Symphony re-dispatches the agent to commit it rather than discarding it, falling back to Blocked after a retry cap (ADR-0014).
_Avoid_: "deploy" (Landing merges a branch; what happens to infra after a merge is separate), "PR" (a PR is one optional Landing shape, not Landing itself)

**Project Scaffold**:
The skill that stands up a new Plane project to match a repo and registers it with Symphony in one pass. It creates the project in the `homelab` workspace from a standard template (states Todo / In Review / Running / Blocked / Done; labels plan / build / approval-required + agent:claude / agent:pi), introspects the fresh per-project state/label UUIDs onto the binding, appends a complete entry to `bindings.yml`, and drops a `WORKFLOW.md` stub for the human to author. Project creation is a live Plane mutation. It does *not* carry the homelab-era domain labels (patrol/security/infra/…).
_Avoid_: "template" (the template is the static shape the scaffold applies; the scaffold is the act of applying it and registering the binding)

**Skill**:
A named, reusable agent procedure the operator can direct an issue toward (e.g. `dev-plan`, `diagnose`, `code-review`). [[Podium]] tracks Skills in three forms: an operator-curated **catalog** of known Skills (populates the UI dropdown), an Issue-level **preferred Skill** the operator names to direct the next dispatch, and a Run-level **invoked Skill** captured from the agent's output as a record of what actually ran. The catalog is non-live — refreshed by an operator command, not auto-discovered each session — and `run.skill_invoked` is advisory free-text, not foreign-keyed to the catalog (an agent may run ad-hoc Skills the catalog has not indexed).

The **preferred Skill is consume-on-dispatch**, not standing config: it is invoked exactly once. The scheduler captures it into the Run's invoked Skill, then clears the Issue's preferred Skill in the same dispatch (only once the Run row is recorded — a blocked or deferred dispatch leaves it intact). A skill-less dispatch is valid and runs as a plain agent continuation (no skill-invoke directive injected). This mirrors a CLI `/skill` invocation: the operator names a Skill once, the next Run uses it, and re-dispatches (e.g. an Operator Reply) run skill-less unless the operator names a Skill again. This is deliberately asymmetric with the other Issue-level dispatch properties (**preferred model**, **reasoning effort**), which are *standing* — they persist across every Run like a CLI session's model choice and are not consumed.
_Avoid_: "command" (a Skill is the named procedure, not its invocation syntax), "tool" (a tool is what a Skill internally calls; the two are not interchangeable)

**Playbook**:
An operator-defined multi-step procedure the operator triggers against an issue (or binding) to run a sequence of actions — the canonical examples are the rralph / tralph pipelines, but Playbooks are not limited to those. The intended primary value of [[Podium]] is letting the operator define and trigger Playbooks once an issue is "on the same page" with the AI. **Distinct from [[Workflow]]**: a Workflow is *static, per-repo, engine-read* on every dispatch (`WORKFLOW.md`); a Playbook is *operator-authored, multi-step, operator-triggered*, lives in [[Podium]], and orchestrates sequences of dispatches (and potentially other actions) rather than instructing a single one. Schema and trigger semantics are deliberately deferred pending the Playbook design pass.
_Avoid_: "Workflow" (the term collision is exactly why Playbook exists as a separate noun; never use Workflow to mean Playbook or vice versa), "pipeline" (overloaded across CI, data-eng, and agent-orchestration; Playbook is the operator-facing term)

**Issue Comments**:
The human-facing operator↔AI communication thread on a [[Podium]] issue, persisted as a markdown blob per issue. Three kinds of writer append to it: the AI coding agent (run summaries + parked questions), the operator (operator [[Reply]] re-dispatch + mid-run [[Steering]]), and [[Temporal]] patrols (`### Patrol (<ts>)` evidence on fail/pass/close). Its purpose is the operator's quick read of an issue: the AI appends a concise summary after each [[Run]] and posts any parked question (see [[Question Park]]) here for the operator to see at a glance. **The unifying primitive is the [[Comment]]: every writer appends an attributed block and nothing else changes; an operator [[Reply]] is the one variant that *also* flips the issue to `todo` (re-dispatch), and a [[Steering]] message is delivered live to the session.** All AI and patrol writes, and operator notes, are plain Comments — append-only, no state flip. The operator may freely edit, delete, or restructure the blob. **The agent does not consume the Comments blob as memory.** On a [[Session Resume]] Run only the newest operator-reply delta is injected; the full blob is re-injected solely on the [[Continuity]] re-feed floor (the rare resume-fail safety net). Its role shifted in 2026-06-Resume from "the continuity mechanism re-fed every Run" to "the human-facing record + the operator's input channel." Distinct from [[Issue Context]] (the AI's own log) and from `description` (the static problem statement filed at issue creation).
_Avoid_: "chat" (Comments are async and operator-curated, not real-time chat), "thread" (overloaded with tmux sessions; "Comments" is the canonical noun), "the agent's memory" (the agent remembers from its native session and [[Issue Context]], not from the Comments blob)

**Issue Attachment**:
An operator-uploaded file queued as context for an Issue's next normally completed [[Run]]. A failed launch, timeout, or nonzero exit leaves it pending for retry; the first successful agent turn consumes it permanently, while files uploaded during that Run wait for the following turn.
_Avoid_: "durable context" (an attachment is one-turn input), "chat attachment" (it belongs to the Issue's next Run, not the Comments blob)

**Comment**:
The append-only primitive on the [[Issue Comments]] thread: an attributed markdown block lands in `comments_md` and nothing else changes — no state flip, no run-state gate. It is what every remote writer needs and the in-process coding agent already had (a plain append). A **Reply** is a Comment that *also* reopens the issue to `todo` (re-dispatch); a [[Steering]] message is a Comment also delivered live to the running session. Over HTTP, `POST /api/issues/{id}/comment` is the Comment door (append, no flip, no gate) and `POST /api/issues/{id}/reply` is the Reply door (append + reopen, gated to a parked/done issue with no active run). Patrols Comment their evidence and reopen/close *explicitly* via a separate state update, instead of borrowing Reply's reopen side effect (the [[ADR-0017]] fix for the C-0281 churn).
_Avoid_: "reply" when you mean a plain Comment (a Reply re-runs the agent; a Comment does not), "note" (operator-facing label for an operator Comment; the primitive is "Comment")

**Issue Context**:
The AI agent's own cumulative log per [[Podium]] issue — a persistent markdown blob the agent writes its full detailed Run output to at completion. This is the **AI-facing** surface (the counterpart to [[Issue Comments]], which is the human-facing one): the agent owns it and does not normally accept operator writes. It has no dedicated operator view in the Podium fly-out — the context tab was removed 2026-06-14 because for `pi_mode: rpc` bindings the agent's stdout (its spoken prose) is the summary block, so `context_md` duplicated the comments thread; the operator inspects per-Run output via the Run Log pane instead. The store itself is unchanged (still the re-feed floor + compaction substrate). It serves the agent two ways: as the agent's own cross-Run **memory log** and as the [[Continuity]] **re-feed floor** (the substrate re-injected when [[Session Resume]] is unavailable); the human's UI read of it is incidental **observability**, not its purpose. Its role shifted in 2026-06-Resume from "the primary cross-Run memory, re-fed every Run" to "the floor's memory": on the common resume path it is **not injected** (the native session holds the verbatim history); it is injected only on fresh/fallback Runs. When Issue Context exceeds a per-binding token threshold, Symphony performs an engine-built compaction step (#026) *before* a fresh/fallback dispatch — rewriting older entries into a single summary block while preserving the most recent N Runs verbatim. Compaction is **skipped on resume Runs** (nothing is injected, so nothing needs trimming) and acts only on `context_md`; the native CLI session manages its *own* context window independently via the agent's auto-compaction — two separate stores, deliberately not reconciled. Compaction is housekeeping: no Run row, no [[Skill]] catalog entry, invisible in the UI except a marker line noting the trim. Distinct from [[Issue Comments]] (the operator thread), from the native CLI session (verbatim, the [[Session Resume]] substrate), and from the per-[[Run]] log on disk (raw stdout/stderr of one dispatch).
_Avoid_: "session memory" (collides with the native CLI session, which is a *distinct* store; Issue Context is the agent-curated blob, not the verbatim session), "transcript" (Issue Context is curated by the agent, not a verbatim capture), "the memory" (post-Resume the native session is the primary memory; Context is the floor + the record)

**Continuity**:
How Symphony lets a re-dispatched issue pick up where the agent left off across [[Run]]s. Two modes exist. **Re-feed** is the stateless mode and the guaranteed floor: Symphony renders the full [[Issue Comments]] and [[Issue Context]] blobs into a fresh prompt every Run, and a brand-new agent process re-reads them. **Session Resume** is the stateful optimization: Symphony resumes the agent's own on-disk session (its verbatim conversation history) instead of re-feeding the curated blobs, sending only the new operator-reply delta. Session Resume is best-effort — when its preconditions do not hold, Symphony falls back to Re-feed for that Run. Re-feed always works; Session Resume is the conversation-quality shortcut layered on top.
_Avoid_: "memory" (overloaded — name the mode), "thread" (already reserved away from tmux sessions)

**Session Resume**:
The stateful [[Continuity]] mode: continuing an issue by resuming the coding agent's persisted CLI session (the verbatim conversation history the agent itself wrote to disk — distinct from the agent-curated [[Issue Context]] and from the operator-curated [[Issue Comments]]) rather than re-feeding those blobs into a fresh prompt. The session is keyed by an id derived deterministically from the issue, and the agent's session store is namespaced by working directory, so resume only succeeds when the issue's checkout directory is unchanged and present. Scoped to the in_review/blocked reply loop: when the operator replies to a parked issue, Symphony resumes that issue's session and sends only the operator's new reply. A resume restores the *conversation*, never the *filesystem* — the working tree may have moved on since the session last ran, which is why Session Resume is gated by a same-agent, same-checkout, session-present, no-git-rug-pull predicate and always degrades to [[Continuity]] Re-feed when any condition fails. Never resumes "the most recent session" implicitly; always the explicit derived id, so a missing session fails loud rather than silently starting empty.
_Avoid_: "resume the worktree" (a session is conversation history, not filesystem state), "session memory" (collides with [[Issue Context]]'s reserved avoidance; "session" here means the agent's native CLI session file)

**Warm Session**:
An issue-scoped Claude tmux session kept alive across multiple [[Run]]s when a binding opts into `claude_persist`. Instead of tearing down the tmux socket after every successful Run, Symphony can reattach to the same parked Claude session and paste the next prompt, preserving Claude's in-memory conversation while avoiding a cold launch and transcript reload. Warm sessions are reaped when the Issue reaches `done` or `archived`, when the parked-idle TTL or max-live cap says they are stale, or at scheduler boot when the single-instance lock confirms this process owns cleanup. Warm Session is Claude-specific; pi uses RPC session resume, not a parked tmux TUI.
_Avoid_: "daemon" or "keeper" (there is no supervised long-lived task; it is a parked tmux session), "Run session" (the lifetime is issue-scoped), "warm worktree" (conversation only, not filesystem state)

**Question Park**:
A [[Run]] outcome in which the agent deliberately ends its turn to ask the operator a clarifying question rather than guessing or completing. It parks the issue in a review/blocked state carrying the question, and the operator's reply resumes the agent's session (see [[Session Resume]]) with the answer, turning the reply loop into genuine turn-taking. This reverses Symphony's prior unattended contract, where the agent was instructed never to ask questions and never to end its turn awaiting operator input. A Question Park is distinct from a blocked-on-error outcome (the agent is not stuck; it is choosing to consult) and from a completed Run (no work is claimed done).
_Avoid_: "blocked" (blocked means impeded; a Question Park is a deliberate consult), "waiting" (too vague — name the outcome)

**Session Tail**:
A live, read-only view of a [[Run]] in progress, produced by tailing the agent's session file (the verbatim conversation history the agent appends to disk as it works) and streaming it to the operator console. Recovers in-flight visibility — tool calls, reasoning, edits as they happen — without changing Symphony's separate-process scheduler architecture, because the session file is a live stream even though the per-Run disk log is only written once at process exit. Distinct from the per-Run log (raw stdout/stderr, finalized at exit) and from [[Issue Context]] (the agent-curated cross-Run summary).
_Avoid_: "live log" (the per-Run log is not live; the session file is), "terminal" (Session Tail is a read-only stream, not an interactive shell)

**Steering**:
A live operator input that redirects a [[Run]] *while it is still working*. For pi RPC bindings, Steering is delivered race-free through pi's RPC `steer` command after the agent's current tool calls finish and before its next LLM call (ADR-0010). For local Claude bindings with `claude_persist: true`, Steering is delivered by tmux send-keys into the issue's [[Warm Session]]; Claude queues that pasted message and applies it at its next turn boundary. Claude abort is separate: it interrupts the current turn with Esc/interrupt semantics rather than appending a next-turn message. Non-persist Claude still uses [[Question Park]] turn-taking instead of live Steering. Distinct from an operator reply, which acts only *between* Runs — a reply flips the issue to `todo` and arrives as the next Run's newest-reply delta, whereas a Steer reaches an already-running Run and does **not** change its state. A Steer is recorded in [[Issue Comments]] as a distinct entry so the operator thread stays a faithful log of everything the operator told the agent; it is not re-injected, because the live session already holds it.
_Avoid_: "reply" (a reply is between-Run and flips state; a Steer is mid-run and does not), "interrupt" when meaning a normal Steer (Claude and pi Steer both land at a turn boundary; Claude abort is the interrupt path), generic "send-keys" as a steering strategy except for the accepted persistent-Claude Warm Session path

## Relationships

- A **Project Binding** maps one tracker project to exactly one repo
- A Podium issue carries a preferred **Skill**; Mode remains a compatibility bridge
- Symphony dispatches an issue to one **Agent**, as one **Run**
- A **Run** executes either in the bound checkout or in an opt-in **Run Worktree**; a global cap bounds how many Runs are live at once
- A finished **Run** may trigger **Landing** when an issue (coding or infra) uses an active Podium worktree
- An issue's history lives in three stores: the **native CLI session** (verbatim, agent-owned, the [[Session Resume]] substrate and the primary memory on the resume path), **Issue Context** (agent-curated cumulative log — the re-feed floor + UI observability), and **Issue Comments** (the operator↔AI thread — human record + operator input channel). On resume only the newest operator-reply delta is injected; the session is authoritative. On the re-feed floor the session is absent and Context+Comments are re-injected — worst case equals pre-Resume behavior.

## Historical

- **[[Workflow]] label self-routing during [[Podium]] migration.** Resolved by Podium: `preferred_skill`, description, Comments, and Issue Context replace Plane label-selected prompt routing for active bindings.
- **[[Mode]] scheduled for removal in the Podium migration.** Resolved by Podium: Skill carries operator work shape; Mode remains only as a compatibility projection.
