# Issue 18 — Fix archive in Podium

## Symptom
"Open an issue, change status or column to archive — nothing happens."

## Investigation (2026-06-14)
Traced the archive flow end to end:

1. **Frontend state chip** (`web/frontend/components/IssueFlyout.tsx`): `ChipSelect`
   for `state` lists all `STATE_KEYS` (incl. `archived`, from `lib/issues.ts` STATES)
   and `onChange` calls `onPatch({state})` → `usePatchIssue` → `patchIssue` PATCH.
   Optimistic update rolls back on a 4xx/5xx — a server error therefore looks like
   "nothing happens" (chip snaps back). No client-side guard drops `archived`.
2. **API** (`web/api/main.py:479,650,1049`): `IssuePatch.state` Literal includes
   `archived`; `patch_issue` handles archive teardown/purge. No server-side rejection.
3. **DB**: the live `podium.db` `issue.state` CHECK previously omitted `'archived'`
   while alembic was stamped past 0004 (stamp-vs-run drift). `state='archived'` →
   `CHECK constraint failed` → 500 → optimistic rollback → "nothing happens". This
   was the root cause.

## Resolution — already landed
The CHECK drift was root-caused and fixed by issue #17's agent earlier on 2026-06-14:
- Migration `0008_fix_issue_archived_check` (commit `b26f31f`) idempotently rebuilds
  `issue` with `'archived'` in the CHECK.
- Applied to the live `podium.db`: `alembic_version = 0008_fix_issue_archived_check`,
  CHECK now lists `'archived'` (verified read-only this session).
- 81 targeted tests pass, incl. `tests/test_alembic_baseline.py::test_upgrade_repairs_stale_archived_check`
  (regression for this exact drift) and `web/api/tests/test_issue_patch.py` archive cases.

Archiving via the flyout **state chip** works once the live DB carries the fixed CHECK,
which it now does. No new code change required for #18 — verification confirms #18 is the
same complaint as #17 and is resolved.

## Genuine remaining gap (not archive-specific)
`web/frontend/components/KanbanBoard.tsx:4-5` — dnd-kit is wired as a placeholder but
**no drag handlers are attached**. "Change the **column** to archive" by dragging a card
does nothing for *every* column, not just Archived. This is an unimplemented feature, not
a regression. Archiving is supported only via the flyout state chip today. If drag-to-column
archiving is wanted, that is separate frontend work.
