# Dispatch Claude through tmux send-keys, not print mode

Symphony dispatches agents heterogeneously: **pi** runs one-shot (`pi --print --no-session`, success from exit code), but **claude** is driven as an interactive session inside a private-socket tmux window, with prompts pasted via `load-buffer`/`paste-buffer` + `Enter` and completion detected by polling `capture-pane` for a per-run nonce Done Marker.

We chose tmux send-keys for claude because Anthropic is removing usable `claude -p` (print/headless) access for this account, so the simpler one-shot path available to pi is not an option. Because a tmux session has no exit code, claude reuses the verdict protocol pi already speaks: a `SYMPHONY_RESULT: done|review|blocked` line (plus optional `SYMPHONY_SUMMARY:`) emitted before the Done Marker, backstopped by post-run side-effect inspection (commit present for `build`, plan artifact written for `plan`). This reuses the existing marker parser the rest of the pipeline already consumes rather than inventing a new vocabulary. The engine is ported from the proven `dev-review-claude` skill rather than invented here.

## Amendment — 2026-06-12

The claude engine was paused when Symphony went thin-engine (the dispatch gate blocks `agent != "pi"`). Before wiring it back in, the completion mechanism is amended: the original capture-pane nonce scraping is replaced by the file-based completion contract that `dev-review-claude` itself evolved to. The prompt instructs claude to write its full output — including the `SYMPHONY_RESULT:` and optional `SYMPHONY_SUMMARY:` lines — to a per-run result file (write the result first, then touch a per-run done file; the adapter's poll requires both the done file present *and* a non-empty result file before declaring completion, so a mid-flush write can never be read half-finished). The adapter polls for that two-condition gate and for session liveness; pane capture is retained only as diagnostics (it becomes the run log's stderr, ANSI-stripped). File-based completion is deterministic and immune to scrollback truncation and marker-echo ambiguity, which pane scraping is not. The side-effect inspection backstop (commit present, plan artifact written) is dropped for parity with the pi path, which never grew one; verdicts come from `SYMPHONY_RESULT` alone.

Because the pane echoes the pasted prompt (and the prompt necessarily quotes the marker vocabulary and may quote WORKFLOW.md text), **all post-run text parsing for claude runs — verdict marker, summary marker, and the permission/approval gate regexes — operates on the result-file content only, never on the pane-derived stderr**. Scanning the pane would let an echoed preamble example or a workflow phrase like "approval required" falsely trip `_hit_approval_gate` or override the authoritative `SYMPHONY_SUMMARY`. Per-run artifacts (result file, done file, tmux socket) are namespaced `symphony-claude-<issue-id>-<nonce>` — distinct from the `drc-*` namespace so a claude run that itself invokes the `dev-review-claude` skill cannot collide with it — and the literal paths are interpolated into the prompt preamble (the run id does not exist yet at the adapter seam, so artifacts key on issue id plus a fresh nonce).

Further decisions locked with the amendment:

- **Engine implementation** is Python-native inside the agent-runner seam, porting the mechanics of `dev-review-claude`'s `engine.sh` (private socket per run, ready-pattern poll, `load-buffer`/`paste-buffer` + Enter, idempotent cleanup) rather than vendoring the bash script. The skill's state-file layer exists only because separate Bash tool calls share no state; a long-lived Python process does not need it, and Python tmux calls are testable with the suite's existing injected-fake style. Wiring is more than unblocking the gate: `RoutingAgentAdapter` currently hardcodes the pi adapter and must gain a claude adapter plus per-call agent resolution from the issue's labels.
- **Permissions**: the claude session runs with bypass permissions. Symphony's safety model is per-binding `WORKFLOW.md` policy plus the review/blocked state machine, not interactive prompts; an unattended permission prompt is just a stall until timeout.
- **Unattended contract**: the adapter wraps the rendered prompt with a preamble stating that nobody will respond — claude must never ask questions or end its turn awaiting input, and if genuinely blocked must write `SYMPHONY_RESULT: blocked` and touch the done file rather than stall. The prompt renderer stays agent-agnostic.
- **Lifecycle** (six mappings): wall clock is the existing `run_timeout_ms`. (1) Ready-pattern poll is bounded (~30s, ported from the skill engine): a session that launches but never shows the bypass-permissions ready footer (auth prompt, onboarding screen, model unavailable) fails fast with a synthetic error instead of burning the full timeout. (2) Done file + non-empty result → exit 0. (3) Session death without done file → synthetic exit 1 with pane tail. (4) Done file with a missing or empty result file → synthetic 137 (mirrors `pi_silent_exit`; missing and empty are treated identically). (5) A non-empty result *without* a `SYMPHONY_RESULT` marker is **not** a failure — it falls through to the default `review` verdict, exactly as a markerless pi run does (true parity). (6) Timeout → kill session, `timed_out=True`. Startup reconcile kills any surviving `symphony-claude-*` tmux sockets, once globally (sockets are not per-binding) — an orphaned session's Run row is already failed by the run reaper, so adopting orphans buys nothing.
- **Model mapping**: claude entries in `models.yml` carry no provider and ignore `reasoning_effort` (advisory tuning, not routing — silent-ignore does not violate the gate's fail-loud rule); the claude dispatch branch passes the catalog id verbatim to `--model` and never appends the `:effort` suffix the pi branch uses. `models.yml` gains per-agent defaults: **at most one `default: true` per agent**, and resolution becomes agent-aware (`resolve_model` receives the resolved agent and selects that agent's default; a missing default for the resolved agent is a loud resolution error, as is an agent/model mismatch such as `preferred_agent=claude` with a pi model). The claude default is `claude-opus-4-8`. Run rows store the provider verbatim — empty for claude, never falling back to the legacy pi provider config. Ripples: the startup pi probe's `resolve_model(None, ...)` call passes `agent="pi"`, and the new-issue form's default-model preselect becomes agent-aware (today it takes the first `default: true` regardless of agent, which would preselect a mismatched pair on an untouched form).
- **Skills**: no `--skill` flag — claude discovers `~/.claude/skills` natively; the preamble directs it to invoke the issue's preferred skill by name.
- **Context compaction stays pi-only**: the pre-Run compaction step parses the agent's stdout as the replacement Issue Context, which is incompatible with the claude adapter's completion-contract preamble. Compaction is engine housekeeping, not issue work — it always dispatches through the pi adapter with the pi catalog default model, regardless of the issue's resolved agent.
- **Startup probe**: a lightweight claude pre-flight (tmux and claude binaries resolve under the service environment, `claude --version` succeeds) runs at boot when any binding can dispatch claude — the analogue of `verify_pi_support`. A full non-interactive auth probe is impractical for subscription auth; an expired login surfaces as a fast ready-pattern failure on the first run instead.
- **Pre-dispatch audit**: before the first live claude dispatch, rendered prompts and each binding's `WORKFLOW.md` are audited for pi-only assumptions (instructions to use the `plane` CLI helper, pi-specific metric markers) that claude cannot satisfy.
