# Replace Plane with Podium, a Symphony-native tracker plus operator console

ADR-0002 generalized Symphony behind a Tracker Adapter seam while keeping Plane as the only tracker implementation, on the reasoning that self-hosted Plane gave privacy and that no ecosystem tool spoke Plane anyway. Eighteen months of operating that pair surfaced a different set of frictions: label-encoded semantics (`mode:plan`, `mode:build`, `agent:pi`, `approval-required`, `approved`, `scheduled`) leak engine behavior into a vocabulary the tracker treats as freeform tags, every engine-significant operator action requires a Plane round trip plus a per-binding UUID map kept in sync with the tracker, comments and AI session logs share one stream so reading either is noisy, and there is no native place for per-Issue operator levers like agent override, model override, or worktree opt-in. Thin engine v2 (commit `e73e924`) has separately deleted worktree-per-Run, the Claude tmux adapter, the plan→build artifact handoff, and the Landing step for coding bindings. The live bindings today are split: `trading` is a `coding` binding (no schedule, no blocked reconciler, no approval gate, no worktree, agent owns git), and `homelab` is an `infra` binding (still exercises schedule, blocked reconciler, and approval-gate code paths — though the worktree mechanism is deleted at the module level for both). ADR-0001 (Claude via tmux) is already inert because thin engine v2 deleted the only adapter that used it. ADR-0003 (worktree-per-Run with global cap) is already inert at the module level — its mechanism is gone regardless of binding type. A separate dispatch-level bug compounds the picture: `scheduler.py:488` keys `is_coding` off `bindings[0].binding_type` alone, so today the whole `is_coding` branch resolves against whichever binding happens to be first in `bindings.yml` rather than per-issue — flagged here for cleanup but out of scope for this ADR. The combined effect is that Plane's data model now sits one layer above what the engine needs and the engine's worktree/landing model sits one layer below what an operator console would want — both sides drifted away from the original architecture.

We chose to build **Podium**, a Symphony-native tracker and operator console living at the repo root under `web/`, and to retire Plane. Podium is FastAPI plus uvicorn on port 8090 serving a Next.js App Router frontend on port 8091. The Symphony scheduler keeps its own `symphony-host.service` unit unchanged; Podium adds two sibling units — `podium-api.service` and `podium-web.service` — each with its own `OnFailure=telegram-alert@%n.service` wiring so the three processes fail and restart independently rather than dragging each other down. The store is SQLite at `/var/lib/symphony/podium.db`, schema evolution via Alembic migrations checked into the repo. Both Podium ports bind to localhost; external access goes through the existing Authelia reverse proxy on 9091. Binding becomes the project — there is no separate Project entity, because the original Plane Project layer existed only because Plane could not address bindings directly. Run becomes a first-class table (one row per dispatch, mutable state machine, log captured to disk at `runs/{id}.log`, no event-log table for v1; on Symphony startup, any row still in `queued` or `running` is reaped to a synthetic `failed`/`blocked` verdict with a `restart-orphan` summary so the board reflects reality after a crash or restart). Issue carries operator intent plus a latest-Run projection (`latest_run_id`, `latest_verdict`, `latest_run_state`, `last_event_at`) so the board reads cheaply without joining. Labels are dropped entirely: every operator-set field that affects AI behavior becomes a typed column (`preferred_agent`, `preferred_model`, `preferred_skill`, `reasoning_effort` default high, `worktree_active` default false, `max_duration_seconds`, `base_branch`). Mode is dropped; **Skill** replaces it as the work-shape lever, structured as a catalog (CLI-refreshed `skill` table) plus a per-Issue `preferred_skill` plus a per-Run `skill_invoked`. Comments and Context coexist as separate Issue blobs: `comments_md` is the bidirectional human↔AI thread (humans write fully, AI writes concise summaries), `context_md` is the AI-only session log (AI writes full output, AI reads on dispatch). When `context_md` exceeds threshold, Symphony invokes the configured agent with a hardcoded compaction prompt — engine-built compaction, not a Skill, zero schema impact. Worktrees come back as opt-in per-Issue persistence: `worktree_active=false` (default) runs the agent in the repo checkout exactly like thin engine v2; `worktree_active=true` creates a persistent per-Issue branch and worktree, and on Issue→Done Podium fast-forward-merges that branch to the base branch and tears the worktree down. A conflict, a diverged base, or a force-pushed base aborts the merge, leaves the worktree intact, and posts a blocked comment for the operator — no merge-commit fallback, no force-push, no rebase. The Tracker Adapter seam from ADR-0002 carries Podium alongside Plane: `bindings.yml` gains a `tracker: plane|podium` field, and the two implementations satisfy ADR-0004's role-based contract by *different mechanisms* — Plane resolves Roles to label/state names plus UUIDs, while Podium projects Roles directly onto typed columns (`mode:*` and `approval-required`/`approved`/`scheduled` fold into columns or disappear, `agent:*` becomes `preferred_agent`, the five states remain as an enum). ADR-0004's role-as-engine-interface stands; its Plane-shaped contract is partially superseded for the Podium implementation. Migration is fresh-stand-up per binding — trading first (it is a proof-of-concept repo with no data worth keeping), then homelab once Podium has been tuned. Plane is archived after both bindings cut over. Authentication is a single bcrypt-hashed shared password sourced from `/home/james/symphony-host.env` as `PODIUM_PASSWORD_HASH` (single-user system, James only). WebSocket pushes Run state transitions and Issue field changes; clients reconcile from row state on reconnect. Test surface is Playwright stories at screen level for the UI and pytest for the FastAPI layer.

The alternatives were rejected for concrete reasons. **Staying on Plane** preserves the round-trip cost, the per-binding UUID maps, the label-as-semantics shape (which was the right lever inside a tracker we did not own but is the wrong lever once we own the store), and the lack of a place to hang per-Issue operator levers — the exact frictions that drove the decision. **Adopting a third-party platform** (sortie, Composio, Warren, Code Conductor, Agent HQ) is the path ADR-0002 already rejected and the calculus has not changed: every mature platform assumes headless Claude (already abandoned here), none speak Plane natively, and adopting any of them would still require writing a custom tracker adapter and a custom agent adapter — the two hardest parts — to gain a UI we are now able to build cheaper. **Adding a UI shell over Plane** (a Next.js frontend talking to Plane's API) would dissolve the label-as-semantics problem visually but not structurally: the database underneath would still be Plane's, every operator lever would still require Plane API writes, and the per-binding UUID maps would still need maintaining. **A two-entity Project + Binding schema** in Podium would re-create the Plane drift Podium is escaping; one entity (Binding-is-Project) is the natural shape now that nothing else writes to the project layer. **An event-log-only Run model** would buy clean audit and replay but pays off only when multi-user replay or post-hoc analysis exists; single-user v1 reconciles from row state on WebSocket reconnect. **Keeping Mode as a tracker concept** while adding Skill alongside would create two parallel work-shape levers; Skill subsumes Mode entirely because Skills are catalog-driven, per-binding customizable, and already where the operator's tuning energy goes. **A data migration tool from Plane to Podium** would burn engineering budget on data neither binding cares about — trading is disposable, homelab will be re-tuned against Podium anyway.

What ADR-0002 said and what stands: the **Tracker Adapter seam** stands — it is the structural reason this swap is cheap, and Podium is its second implementation. The **Agent Adapter seam** stands — pi-only after thin engine v2, but the seam is the long-term hedge. The **Workflow concept** (`WORKFLOW.md` per binding) stands unchanged. The validation framing — that sortie independently converged on Symphony's architecture — stands; Podium's data model is the natural narrowing of that same shape to a single-user, single-tracker case. What ADR-0002 said that this ADR overrides: the conclusion to **"stay on self-hosted Plane now for privacy"** is superseded; Podium is also self-hosted (same host, same network), removes the Plane process entirely, and removes the per-binding UUID map maintenance burden that Plane required. ADR-0003 (worktree-per-Run with global concurrency cap) is not reversed by this ADR — thin engine v2 already deleted ADR-0003's worktree mechanism at the module level — but Podium's `worktree_active` opt-in per-Issue persistent worktree is the lever that reintroduces worktrees as an *operator-controlled* feature rather than an *engine-default* one, which is the meaningful shift in posture. ADR-0001 (Claude via tmux send-keys) is not reversed by this ADR — thin engine v2 already deleted the ClaudeAgentAdapter — but Podium's `preferred_agent` column keeps the door open for re-adding a Claude adapter against the same seam if the headless story ever changes.

The accepted costs are real. Symphony grows two new systemd units (`podium-api.service`, `podium-web.service`), a Node and browser toolchain to support the Playwright screen-level tests sitting alongside Python's pytest, and a SQLite file at `/var/lib/symphony/podium.db` whose loss erases Run history, Issue Context, and Comments — so that file is added to the existing host backup chain (rsnapshot of `/var/lib/symphony/`) and the absence of off-host replication is acknowledged as the single-user posture's known weakness. Run logs under `runs/{id}.log` are retained on a rolling 90-day window and capped at the most recent 100 per Issue, whichever cuts first. Single-shared-password auth bakes the single-user assumption into the system; multi-user is an explicit non-goal, and the bind-to-localhost-behind-Authelia posture means the auth weakness is bounded to a host-LAN attacker who is already past the reverse proxy. The Plane-coupled skill suite migrates rather than survives unchanged: `symphony-project-scaffold`, `symphony-binding-smoke`, `symphony-bindings-status`, `symphony-plane-recover`, and `symphony-onboard-project` either retire (Plane-only) or grow Podium counterparts; `symphony-workflow-author` becomes tracker-agnostic via the Tracker Adapter seam since it edits `WORKFLOW.md` rather than the tracker. Migration is a clean cutover per binding rather than a gradual coexistence, which means a brief window per binding where neither tracker is authoritative — acceptable here because trading is disposable and homelab is operated by exactly one person. The schema rewrites WORKFLOW.md tuning as migration work: anywhere a workflow currently routes on label semantics, that routing has to become column semantics. CONTEXT.md retires Mode, Run Worktree's Plane-era framing, and the Landing-via-merge story when Podium ships; this ADR is the prompt for those edits. The reward is that every engine-significant operator action becomes a typed write to a local SQLite row, every Run becomes a first-class entity with cost, tokens, branch, and verdict captured, and the operator surface becomes a place where new levers can be added by adding a column rather than by inventing another label and another UUID map.
