# Continue parked issues by resuming the agent's CLI session, best-effort, over a re-feed floor

Symphony's continuity model has been deliberately stateless. pi was dispatched one-shot (`agent_runner.py:107` `--print --no-session`), so no session existed to resume; an operator reply flipped the Issue back to `todo` and the next Run re-rendered the full `comments_md` (operator thread) and `context_md` (agent-owned cumulative log) into a fresh prompt that a brand-new process re-read (`prompt_renderer.py`, operator-reply path). #026 even built engine-owned compaction *on top of* that stateless store. The model is robust — it survives restarts, separate-process scheduling (ADR-0006), and worktree teardown — but it is lossy on iteration: every follow-up hands the agent a curated summary of its own past, not the reasoning, dead-ends, and file reads it actually accumulated, so the agent visibly re-derives context each reply and the operator feels it getting dumber. The goal that motivated this decision is continuity *quality*: make a Podium issue feel, on the review-reply loop, like sitting at the agent CLI without leaving.

We decided to continue a parked Issue by **resuming the coding agent's own on-disk CLI session** rather than re-feeding the curated blobs — but only as a **best-effort optimization layered over the stateless re-feed, which remains the guaranteed floor.** Both agents support it on the CLI we already drive: pi via `--session-id <id>` (create-or-resume in one flag) and Claude via `--session-id <uuid>` on the first run and `--resume <uuid>` thereafter, the branch chosen by probing whether the session file exists. The session id is **derived**, not stored — `UUIDv5(namespace, issue.id)` — so the Issue *is* the session key and nothing can drift from disk. Resume only pays off if it lets us stop re-feeding, so on a resume Run the prompt collapses to the per-Run mechanical wrapper plus the single newest operator-reply block; the Issue body, full Comments, full Context, and WORKFLOW.md are omitted because the session already holds them. Symphony keeps *writing* `comments_md`/`context_md` (for the UI and the fallback floor) but stops *injecting* them on resume, and skips the #026 compaction step on resume Runs since there is no injected Context to compact.

Because a session persists the *conversation, not the filesystem*, and is namespaced by working directory, blind resume would sometimes hand the agent a mental model that no longer matches the tree. Resume is therefore gated by a four-part predicate, all of which must hold or the Run falls back to fresh + full re-feed: same agent kind (a pi session is not a Claude session), the cwd still exists and is unchanged (its encoded-cwd slug must match), the session file is present, and git HEAD has not moved under the session since it last ran (`run.agent_session_sha`, captured at Run start, compared at the next dispatch). This confines resume to the **in_review/blocked → operator-reply loop**, where the worktree is intact and nothing has merged; the Done-reopen case and every predicate failure use re-feed, and the worktree lifecycle (#021 FF-merge + teardown on Done) is left untouched. The only new persistence is two columns on the `run` row — `agent_session_sha` and a `resumed` boolean (observability only). Resume never uses `--continue`/`-c` ("most recent session in cwd"), which silently starts empty on a mismatch; an explicit derived id makes a missing session fail loud, and a runtime resume error is caught and re-fed within the same tick with loud `resume_skipped`/`resume_failed` markers, in keeping with Symphony's silent-failure guardrail.

The alternatives were rejected for concrete reasons. **Capturing the agent-minted session id and storing it** — the idiomatic Claude approach — reads `session_id` off the Agent SDK's result message; Symphony's Claude path is the tmux send-keys CLI (ADR-0001), not the SDK, so capture would mean either a full SDK migration that retires the just-hardened tmux adapter (#042–#046) with no pi-side equivalent, or a fragile filesystem scrape of the newest `.jsonl`. Deriving the id sidesteps both and stays symmetric across pi and Claude. **A dedicated `issue_session` pointer table** was rejected as a second source of truth that can disagree with disk while storing a value that is already a pure function of the Issue. **Keeping worktrees alive after merge** so Done-reopen could resume was rejected as fighting the #021 lifecycle for the rarest case; re-feed covers it. **Letting native session auto-compaction replace #026** was deferred, not adopted — the engine compaction and the native transcript are two context stores whose reconciliation is a separate, thornier decision. Anthropic's own SDK docs, notably, recommend re-feeding results over "shipping transcript files around" for ephemeral/cross-host situations, which is exactly why re-feed stays the floor rather than being removed.

The accepted costs: continuity behavior now depends on hidden filesystem state (the session files under `~/.claude/projects` and `~/.pi/agent/sessions`), so those stores need retention/cleanup and their loss silently downgrades an Issue to re-feed (correct, but a quality regression the operator may not notice); WORKFLOW.md edits made while an Issue is parked are invisible to a resume Run until it falls back; and pi↔Claude agent swaps, model-incompatible resumes, and any base-branch advance all force re-feed. All are bounded by the re-feed floor: the worst case is today's behavior. The decision is reversible per-Issue and globally — disabling resume reverts to pure re-feed with no schema rollback required, since the two `run` columns are nullable and unread by the fallback path. This ADR records why a system that deliberately chose stateless continuity now resumes native sessions for one specific loop, and why that resume is always a best-effort layer and never the floor — so the next reader does not "simplify" by deleting the re-feed path or by trusting resume unconditionally.

**Resolved 2026-06-13 (the deferred compaction-reconciliation decision):** the two context stores stay orthogonal — there is nothing to reconcile. #026 engine compaction owns `context_md` (the re-feed floor blob) and runs only on fresh/fallback dispatches, immediately before the floor injects it; it is skipped on resume Runs (nothing is injected, so nothing needs trimming). The native CLI session manages its *own* context window independently (pi via RPC `set_auto_compaction`/`compact`, ADR-0010). Two compactors, two stores, no shared authority — so they cannot fight. A corollary role reframe lands in CONTEXT.md, and the cleanest statement of it is an ownership split: **Issue Comments is the human-facing surface; Issue Context is the AI-facing surface.** Comments exists for the operator — the agent writes per-Run summaries and parked questions there *for the human to read*, and the operator writes replies and Steers there; the agent never consumes the Comments blob as memory (on resume only the newest operator-reply delta is injected; the full blob is re-fed solely on the re-feed floor). Context is the agent's own surface — its cross-Run log and the re-feed floor substrate; the human may observe it in the UI, but it is the agent's, not the human's. On the common resume path the native session is the primary memory and Context is not injected at all; Context is the floor's memory, not the resume path's. New term **Steering** (ADR-0010) is recorded in Issue Comments as a distinct entry but never re-injected.
