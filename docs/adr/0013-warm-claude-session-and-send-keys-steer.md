# Keep the Claude tmux session warm for the whole Issue, and steer it live via tmux send-keys

The operator goal is the same one ADR-0010 chased for pi — make a Podium Issue feel like sitting at the agent CLI: watch it work, and redirect it while it is still working — but for **Claude**, which ADR-0010 left on park-and-reply only because the sole live-steer mechanism available to it (tmux send-keys) was classed as race-prone (`C-0174`/`C-0176`). Two reframings make the Claude path viable without reversing ADR-0010's safety reasoning.

## What is actually new vs. what already exists

Claude already dispatches through tmux (`claude_runner.py`, ADR-0001): `tmux new-session -d` runs the interactive `claude` TUI, the prompt is pasted via `load-buffer`/`paste-buffer` + `Enter`, completion is a done-file/result-file contract, and the session *transcript* already persists across Runs via a deterministic per-Issue `--session-id` with `--resume`. Two things are new here:

1. **Issue-scoped session lifetime.** Today the tmux session is **run-scoped** — created at dispatch, torn down in the `finally` of `run_claude_agent` (`claude_runner.py:565`). A second Run on the same Issue (resume after `in_review`, blocked retry, sha-drift refeed) pays a cold restart: `new-session`, the ~30s `claude_ready_timeout` wait (`claude_runner.py:467`), and a full `--resume` transcript reload. The decision is to make the session **issue-scoped**: a completed Run leaves the tmux session alive and Claude parked idle at its prompt; the next Run **re-attaches** to the live socket and pastes only the new prompt/delta, skipping `new-session`, the ready-wait, and the cold reload.

2. **Live steer for Claude via send-keys, made safe by Claude's own message-queue.** The operator types a steer in Podium; Symphony delivers it to the running session with the *same* `load-buffer`/`paste-buffer`/`Enter` mechanism `_paste_and_submit` already uses (`claude_runner.py:687`) — exactly as if a human attached to the session and typed a prompt. Claude's TUI queues a message typed mid-turn and acts on it at its **next turn boundary**. So Symphony does not need to find a safe injection point itself; Claude's TUI provides the boundary. This sidesteps the `C-0174` race ADR-0010 feared — that race was about *Symphony* having to interleave keystrokes with a running tool call; here Symphony just types and Claude's TUI sequences it. The mechanical paste/Enter reliability (`[Pasted text …]` placeholder absorbing Enter) is already handled by `_paste_and_submit`'s retry loop and is reused verbatim. `bypassPermissions` (`claude_runner.py:434`) guarantees Claude is never parked at a permission gate, so typed text always lands in the input box.

## Decision

- **Claude tmux sessions are issue-scoped, warm across Runs.** A Run that ends does **not** tear down tmux; it leaves Claude idle at the prompt. `run_claude_agent` is split so dispatch checks an issue→socket registry: live socket present → reattach + paste; absent or dead → cold `new-session` + `--resume` (the current path, now also the reattach-failure fallback). Reattach failure never blocks a Run.

- **Teardown triggers (belt-and-suspenders):** the session is reaped when (a) the Issue reaches a **terminal state — `done` or `archived`**; or (b) an **idle-TTL backstop** (~45 min with no transcript activity) fires for a session whose Issue is wedged (e.g. stuck `in_review` awaiting a human who never returns); or (c) the **boot/periodic orphan sweep** finds a session with no matching live Issue. This generalizes the existing `reap_orphan_claude_sockets` (`claude_runner.py:118`) from per-run-pidfile ownership to issue-liveness ownership.

- **Live steer is extended to Claude over the existing steer channel.** The ADR-0010 #056 steer queue (`web/api/steer_queue.py`) is already agent-agnostic (keyed by `run_id`, kinds `steer`/`abort`). The API's explicit Claude reject (`web/api/main.py:1263`) is dropped for persistence-enabled Claude bindings; the flyout steer affordance (#057) is un-gated for them. `run_claude_agent`'s poll loop reads the queue for its Run and delivers each `steer` via `_paste_and_submit` (Claude queues it for its next turn); `abort` maps to Esc/interrupt or session kill. `run_id` is threaded into `run_claude_agent` so it reads the right queue file. **Steer semantics (spike-confirmed): a queued steer APPENDS a next turn — it does not preempt the turn in flight.** A steer delivered while Claude is mid-turn takes effect only after that turn finishes (Claude may complete work the operator meant to redirect). Operators who need "stop what you're doing now" use `abort` (Esc) first; the UI should make the two affordances distinct so the append-vs-interrupt difference is not surprising.

- **Unsupervised session model.** A warm session is a **parked tmux**, not a supervised long-lived task. Live steer is delivered only **during an active Run** (Claude working — the common single-Run case, plus any resume Run). In the `in_review` gap between Runs there is no Symphony loop pumping the queue; the operator uses **park-and-reply** (an Issue comment flips the Issue to `todo` and resumes — ADR-0009), which now lands in the *warm* session via reattach. So: steer = redirect a working Run; comment = wake a parked Issue. Both reach the same live Claude session.

- **Per-binding opt-in.** A binding flag (e.g. `claude_persist: true`, mirroring `pi_mode`) enables warm sessions + Claude steer. Default is the current per-run teardown, which stays the rollback path.

- **Concurrency.** A warm but idle session (Issue `in_review`, no active Run) does **not** hold a `run_cap` semaphore slot; only an active Run does. A separate max-live-sessions safety cap bounds resident sessions; the idle-TTL does the real bounding given Issues are short-lived in practice.

## Considered options

- **Idle-gated steer delivery (Symphony waits for a parked pane before send-keys).** Rejected as unnecessary: it reproduces the nudge's idle detector to find a safe boundary Symphony does not need to find, because Claude's TUI already queues a mid-turn message and acts on it next turn. Delivering immediately and letting Claude sequence it is simpler and matches what a human attached to the session would do.
- **Interrupt-now as the default (Esc, then paste).** Rejected as the default: Esc mid-tool-call is less predictable and can leave a half-state. Kept as the explicit `abort` action only.
- **Supervised session (a long-lived keeper task per live session).** Rejected: it would allow steer during the `in_review` gap too, but that case is already served by park-and-reply, and a keeper adds a new first-class lifecycle to build, test, and reap. Unsupervised parked-tmux is the minimal change.
- **Programmatic mid-run send-keys without relying on Claude's queue (the ADR-0010 race).** Rejected — that is exactly the `C-0174` class ADR-0010 ruled out. The viability here rests entirely on Claude's TUI queuing the typed message, which is the spike gate below.

## Consequences

- ADR-0010's "live steer is pi-only / Claude is park-and-reply only" is **amended**: Claude gains live steer, but via a different mechanism (send-keys into Claude's own message-queue) and a different safety argument (Claude's TUI provides the turn boundary, not Symphony). pi's RPC `steer` remains race-free-by-construction; Claude's is next-turn-by-construction. Both live under the one Agent Adapter capability interface.
- ADR-0001's tmux session moves from run-scoped to **issue-scoped**; the done-file/result-file completion contract and `_paste_and_submit` race handling are unchanged and reused.
- New operational surface: warm sessions resident across the `in_review` gap (memory, not tokens), a reattach path, an issue→socket registry, an idle-TTL reaper, and a max-live-sessions cap — the warm-session analogue of ADR-0010's RPC lifecycle slice.
- `CONTEXT.md` glossary updates land with the implementation, not before (it is a live-infra glossary agents read as current behavior): the **Steering** term gets a Claude clause (and its `_Avoid_: "send-keys"` line is revised — send-keys is now the accepted Claude steer mechanism), and a new **Warm Session** (issue-scoped Claude tmux) term is added. `wiki/CLAIMS.md` records the amendment to `C-0176`/ADR-0010.
- **Deployment contract (added post-soak):** the steer channel is split across two processes — `podium-api.service` writes the queue, `symphony-host.service` reads it — so both must resolve `$SYMPHONY_RUNTIME_DIR/steer` to the same filesystem path. Because symphony-host runs `PrivateTmp=yes`, the default `/tmp/symphony` is per-service-namespaced and the two never share it; live steer/abort then silently no-ops. Both units must carry `SYMPHONY_RUNTIME_DIR=/run/symphony` (or another shared, non-`PrivateTmp` path). This is a unit-config invariant, not enforced in code.
- Gated on two spikes before build: (1) Claude TUI reliably queues a message typed mid-turn via send-keys; (2) a second Run reattaches to a live socket and pastes without a cold `--resume`. If (1) fails, Claude steer falls back to idle-gated or stays park-and-reply; if (2) fails, warm-reattach is dropped and only teardown-deferral/observability remain.
