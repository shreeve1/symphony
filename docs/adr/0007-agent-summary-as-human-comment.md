# Agent's natural turn-end summary becomes the human comment; machine state moves off the comment stream

Status: accepted

The Podium `comments_md` thread is meant to be the bidirectional human↔AI surface (ADR-0005), but in practice every dispatch wrote three layers of machine noise into it: a `Symphony claimed at <ts>` claim comment, a pipe-delimited `**Timeline** — claimed: … | finished: … | agent: Xms | sha: …` line appended to every closing comment, and a `### Symphony AI Summary` H3 wrapper prepended to each block by `tracker_podium.post_comment`. The human-readable content itself was a single terse line: agents were instructed (in `_wrap_prompt` and in each binding's `WORKFLOW.md`) to emit one `SYMPHONY_SUMMARY: <one short sentence>`, and `_extract_summary`/`_parse_summary_marker` collapsed it to one line capped at 500 chars. The operator wanted the opposite shape — the comment tab should carry the agent's natural end-of-turn message (what it did, findings, and any questions for the operator), the way an interactive turn ends, not a forced one-liner buried under machine metadata.

We decided to make the agent's multi-line natural summary the comment body and to remove machine state from the comment stream entirely. Agents now emit a delimited `SYMPHONY_SUMMARY_BEGIN … SYMPHONY_SUMMARY_END` block, which Symphony posts verbatim (markdown preserved) as the terminal-state comment for review, done, *and* blocked alike — blocked falls back to the bounded stderr summary only when no block is present. The body is whole-block secret-redacted, has `SYMPHONY_RESULT` and `SYMPHONY_*` metric marker lines stripped from display, and is bounded at ~4000 chars with a head+tail truncation notice on overflow; the fallback chain is block → legacy single-line `SYMPHONY_SUMMARY:` → "Agent finished without a summary." The claim comment is dropped: `_claimed_at` now reads `started_at` from the latest running Run record, keeping comment-parsing only as a fallback for adapters without a run store (Plane). The `**Timeline**` line is dropped from all closing comments — its claimed/finished/duration/verdict/sha data already lives in the Run record. The `### Symphony AI Summary` wrapper is removed from `post_comment`. `context_md` and its `### Symphony Context Append` wrapper are deliberately left unchanged; the AI-only session log is fine as-is.

The output contract is centralized rather than duplicated. Today the marker contract was taught in three+ places across three repos — `_wrap_prompt` (claude), and the `SYMPHONY_SUMMARY`/`SYMPHONY_RESULT` boilerplate in `/home/james/homelab/WORKFLOW.md` and `/home/james/trading/crypto-trading-agents/WORKFLOW.md` (pi reads only the latter) — which would all have to change in lockstep and would drift via the `symphony-workflow-author` scaffold. Instead the contract moves to a Symphony-owned footer appended by `prompt_renderer` to every rendered prompt, so both runners receive identical instructions from one version-controlled source; the per-binding WORKFLOW.md boilerplate and the scaffold template are reduced to a pointer. The accepted costs: the contract leaves WORKFLOW.md where the operator currently reads it, the WORKFLOW.md edits are separate commits in external repos, and historical comments on existing issues keep their old formatting (no backfill) — the `CLAIM_PREFIX` filter in `_fetch_issue_comments` stays to keep legacy claim comments out of re-injected context harmlessly.
