# Dispatch pi via its RPC protocol to enable live mid-run steering; Claude stays park-and-reply on tmux

The operator goal is continuity *quality* taken one step past ADR-0009: make a Podium issue feel like sitting at the agent CLI — explore, watch the work stream, ask the agent things, **and redirect it while it is still working**, not only between Runs. ADR-0009 + issues 047–055 deliver the *async turn-taking* form of that — the agent parks (`SYMPHONY_QUESTION`, #052), the operator replies, the session resumes (#050/#051), and the run is observable after the fact (Live Session Tail, #053) — entirely on the current dispatch shapes (pi `--print`, Claude tmux). But that backlog **deliberately deferred live mid-run steering** ("Deferred (no issues): live tmux send-keys mid-run steering (Claude-only)", `wiki/concepts/session-resume-continuity.md`, `C-0176`) because the only mechanism available for it at the time — injecting send-keys into a running tmux pane mid-tool-call — is the same race-prone class (`C-0174`: paste/Enter races, done-before-result races) that made the Claude tmux path fragile. Park-and-reply was the safe substitute; live steer was the thing nobody wanted to own.

Two facts, verified this session, change that calculus. First, **pi ships a native headless RPC protocol** (`pi --mode rpc`, `docs/rpc.md`): JSON commands over stdin, JSONL events over stdout. Its `steer` command is **race-free by construction** — a steering message is delivered *after the current assistant turn finishes executing its tool calls, before the next LLM call* — which is exactly the mid-run interjection point that send-keys cannot hit safely. It also gives deterministic completion (`agent_end`), clean cancellation (`abort`), a full event stream (`message_update`, `tool_execution_*`, `turn_*`) that *is* a live tail, and native sessions (`--session-id`, `switch_session`, `fork`). Second, **Claude has no such path for this account**: `claude --output-format stream-json` is gated behind `--print`, and `claude -p` headless access was removed for this subscription (ADR-0001, ADR-0002, `C-0016`), so Claude's only authenticated surface remains the interactive TUI driven by tmux send-keys. The asymmetry is permanent under the current account: pi has a clean headless protocol, Claude structurally does not.

We therefore decided **not** to chase a single dispatch mechanism for both agents — neither "pi onto tmux to match Claude" (which would drag the proven, 700+-test pi path onto the fragile mechanism and re-thicken the engine thin-engine-v2 intentionally thinned) nor "both onto a streaming protocol" (impossible: Claude has no headless protocol). Instead we **standardize the Agent Adapter *capability* interface, not the transport**: the engine asks an adapter for completion, tail, and steer; each adapter satisfies those in whatever transport its agent allows. Concretely: **pivot pi's dispatch onto RPC** and build live mid-run steering on the RPC `steer` command, and **leave Claude on tmux with park-and-reply continuity**. Live mid-run steering is, and is documented to be, a **pi-only** capability; Claude's interactivity stops at the park-and-reply turn-taking ADR-0009 already specifies.

The one genuinely new piece of architecture is a **control channel from the web/API process to the live agent process**. ADR-0006 made the scheduler a separate process from the web/API and surfaced engine state by polling *outward* only; live steer is the inverse — operator input flowing *inward* to a running subprocess whose stdin the scheduler holds. We extend, rather than overturn, ADR-0006 by reusing the seam #054 already establishes: where #054 has the web write a wake *sentinel* the scheduler polls, live steer has the web write **steer payloads** to a per-run queue (a file under the runtime dir, e.g. `/run/symphony/steer/<run_id>.jsonl`, or a DB-backed queue — shape TBD) that the pi adapter's event loop polls and forwards as RPC `steer` commands. This keeps the two processes decoupled, restart-tolerant, and free of any new long-lived socket daemon; the cost is poll-interval latency, which is sub-second and acceptable. The synchronous adapter contract (`AgentAdapter.__call__ -> AgentResult`) becomes an internal event-pump loop — spawn, send the initial `prompt`, pump events for tail while polling the steer queue for input, until `agent_end` — but still returns one `AgentResult` whose `stdout` is the final assistant text, so the downstream verdict scrape (`SYMPHONY_RESULT`, `scheduler.py:233`), summary, and metrics parsing are unchanged.

This pivot **re-sequences the ADR-0009 backlog rather than adding to it.** The RPC-agnostic foundation stays as-is and as-built: #047 (run-table columns), #048 (`session_continuity.py` — derive id, eligibility predicate, session-file path; RPC also keys on `--session-id` and session files), and #049 (delta-only renderer) apply unchanged. But **#050 (pi resume e2e) and the pi sides of #052 (Question Park) and #053 (Session Tail) must be rebuilt on RPC instead of `pi --print --session-id`** — RPC provides resume (`--session-id`/`switch_session`), tail (events), and park (`agent_end`/`extension_ui_request`) on one mechanism, so implementing them on `--print` first and adding RPC later would build pi's continuity stack twice. Claude's resume (#051) keeps its `--print`-less tmux/`--resume` design. The work decomposes into independently shippable slices: **A** pi RPC adapter run-to-completion parity (in-app testable, de-risks everything); **B** live tail to the WS hub (overlaps #053); **C** the steer channel (web endpoint + per-run queue + adapter poll/forward + `abort`); **D** the frontend steer box + tail panel (overlaps #053); **E** lifecycle (run timeout enforced across a long-lived process, orphan/queue reaping analogous to the Claude socket reaper, concurrency-cap interaction, a run→channel registry).

## Considered options

- **pi → tmux, to match Claude (the original handoff hypothesis).** Rejected: it standardizes on the *fragile* mechanism, putting the `C-0174` race class (paste/Enter, done-before-result, ready-poll banner scrape, orphan sockets, `PrivateTmp` observability) onto the currently-robust pi path, and re-thickens the engine thin-engine-v2 deliberately thinned. One mechanism, but the wrong one.
- **Async turn-taking only; no RPC (ship 047–055 as designed).** Rejected for *this* goal: park-and-reply cannot redirect an agent mid-tool-loop — it can only act at park points between Runs. It remains the right design for Claude and the guaranteed floor for pi, but it does not deliver live steer, which the operator explicitly wants.
- **Standardize both agents on a JSON streaming protocol.** Impossible under the account constraint: Claude's `stream-json` requires `--print`, which is removed for this subscription. There is no symmetric headless target.
- **Direct socket/IPC between scheduler and web for the steer channel.** Rejected in favor of the file/DB-queue: a bidirectional socket is a new long-lived coupling that fights ADR-0006's deliberate process decoupling and adds a failure mode (dead socket, reconnection) for sub-second latency we do not need.

## Consequences

- pi and Claude diverge further: two transports (RPC vs tmux send-keys) under one Agent Adapter *capability* interface. This is accepted and already the project's direction — the seam exists precisely so agents can differ.
- ADR-0006's input direction is extended: the web process can now write payloads inward to a running engine subprocess via a polled per-run queue. State still surfaces outward by polling/tail; this adds the inward path #054 began.
- The thin-engine-v2 "pi dispatches one-shot" posture is partially reversed for interactive issues. pi one-shot may remain the path for non-interactive/cheap dispatches if a dual-mode split is later wanted; that is left open, not decided here.
- Live mid-run steering is **pi-only**. Issues routed to Claude get park-and-reply, not live steer. Operator routing should bias exploratory/steer-heavy issues to pi.
- New operational surface: long-lived RPC processes (orphan reaping, run-timeout enforcement across the pump loop, concurrency-cap accounting, steer-queue cleanup) — the RPC analogue of the Claude socket reaper/probe.
- A pre-commit parity spike is required and this ADR is gated on it: confirm `pi --mode rpc --skill <dir>` loads the preferred skill (`agent_runner.py:263`), honors CLAUDE.md/AGENTS.md context, and resolves provider/model identically to `--print`. If parity fails, the dispatch-mechanism assumptions here must be revisited before Slice A.
- `C-0176`'s "deferred (no issues)" live-steer item is un-deferred for pi and becomes Slices A–E; the wiki concept page and CLAIMS must be updated to reflect that live steer is now in-scope via RPC for pi and remains deferred (and likely permanently park-and-reply) for Claude.
