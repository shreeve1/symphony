# Remote Bindings dispatch over SSH-exec, not a remote daemon

## Context

Symphony's engine (tracker, scheduler, prompt rendering, verdict scrape, continuity) runs centrally on the aidev host and executes an issue by shelling out to a **local** agent subprocess — `pi` (`--print --no-session`, or `--mode rpc` over the process's stdin/stdout) or `claude` (tmux send-keys) — with `cwd` set to the binding's local `repo_path` (`agent_runner.py`, `claude_runner.py`). Issue #27 asks to run sessions against repos that live on *other LAN/tailnet systems* (the first target is `itadmin@100.95.224.218`, hostname `n8n`, which already has `pi`, `claude`, `git`, and `tmux` installed and reachable by SSH key).

## Decision

A **Remote Binding** is an ordinary Project Binding carrying an optional `remote:` block (`host`, `user`, ssh identity); its `repo_path` then denotes the checkout path *on that host*. Dispatch is performed by a `RemoteAgentAdapter` behind the existing `AgentAdapter` Protocol / `RoutingAgentAdapter` seam: it runs the same pi RPC command shape as local coding bindings, but over `ssh user@host <agent> ...`. SSH supplies exactly what dispatch needs — remote command, remote `cwd`, env forwarding, and a stdin/stdout pipe pair for `pi --mode rpc`, steering, and session resume. The engine stays centralized: one brain, many hands. **No Symphony component is deployed to the remote** — the only things on the remote are the agent CLI, a repo checkout, an SSH key, and whatever sudoers policy the operator chooses to grant.

## Considered options

- **Deployable remote daemon** (the operator's first framing — a "symphony-remote" copied onto each device). Rejected: pi/claude are already self-contained CLIs, and a daemon would re-implement process management, auth, lifecycle, and a wire protocol that SSH already provides — more code and more attack surface for no gain on a trusted LAN/tailnet.

## Consequences

- **Tracker callback path is the load-bearing detail.** Agents report back to the tracker over HTTP via `SYMPHONY_PLANE_API_URL`, which is **loopback-only** (`http://127.0.0.1:8000`; the Podium API binds `127.0.0.1`, only the Next.js web UI is LAN-bound on `10.20.20.16:8091`). A remote agent therefore cannot reach the API directly. Recommended mechanism (pending operator confirmation): the `RemoteAgentAdapter` opens the SSH session with a **reverse tunnel** (`ssh -R 8000:127.0.0.1:8000 ...`) and forwards `SYMPHONY_PLANE_API_URL=http://127.0.0.1:8000` into the remote env, so the remote agent's loopback writes tunnel back to aidev's loopback API. This keeps the API loopback-only — no new LAN exposure, no change to its auth surface. The rejected alternative is binding the Podium API to the LAN/tailnet, which widens exposure for every host.
- **Remote-binding invariants, enforced at config load.** A `remote:` binding must be `type: coding` + `pi_mode: rpc` + `default_agent: pi`; `config._binding_from_mapping` raises `ConfigError` otherwise. `coding` skips local `WORKFLOW.md` rendering and build-mode plan-file validation; `rpc` gives remote bindings the same pi dispatch substrate as local coding bindings; `pi` is the only remote agent `RoutingAgentAdapter` supports. The invariants make the remaining local-only `repo_path` touchpoints unreachable for remote bindings rather than needing per-callsite guards.
- **Local coding parity is the remote standard.** Remote pi dispatch uses SSH-piped `pi --mode rpc`, not `pi --print`, so preferred Skill invocation, Session Resume, and Steering follow local coding-binding semantics. Preferred Skills are shipped per dispatch by copying the selected local skill directory into the remote temp dir and passing the remote `--skill` path. Session Tail still tails an on-disk session *file* that lives on the remote, so a remote tail reader may need `ssh host tail -f` if local parity requires live tailing.
- **Process-lifecycle reapers are local-first.** RPC pidfiles (`/tmp/symphony`) track the local SSH process for remote RPC runs; killing the SSH process group closes the channel and should SIGHUP the remote pi process. Remote orphan sweeps for any process that survives channel close remain a v2 concern because `/proc` liveness checks are host-local.
- **sudo/privilege stays the remote's responsibility**, consistent with ADR-0011 (safety is the repo's/host's job; Symphony stays narrow). Symphony runs the agent as the SSH user and inherits whatever that user can sudo.
- **Terminology:** the concept is a **Remote Binding** (a binding attribute), deliberately *not* a "symphony-remote agent" — there is no Symphony agent on the remote to name.
- **Host badge (operator-confirmed):** because the Podium `binding` table stores only `name`/`display_name`/`color`/`sort_order`/`archived` and the `remote:` block lives only in `bindings.yml`, a Remote Binding is otherwise visually indistinguishable from a local one in the dashboard. v1 therefore surfaces a small **host badge** (e.g. a `⇄ n8n` chip) on the binding card / fly-out so remote execution is legible to the operator.

## Amendment 2026-06-19 — gap walkthrough (growing-fleet hardening)

Remote bindings are a **growing capability** (more hosts over time), not a solo `n8n` demonstrator. A `/grill-me` walkthrough of the dispatch path surfaced gaps; dispositions:

- **Concurrency isolation — serialize per remote binding (implemented 2026-06-19).** Worktrees are inert for remote, so the agent commits directly in `repo_path`, and `run_cap` is **per binding** (live unit `SYMPHONY_RUN_CAP=3`). A single remote binding could therefore put up to `run_cap` `pi` processes in one remote working tree → corruption + tangled commits. Implemented as a **per-binding dispatch semaphore sized to 1 for `binding.is_remote`**: a pure helper `_effective_run_cap(config, binding)` returns `1` when the binding is remote else `config.run_cap`, and `_new_dispatch_state` builds the `_DispatchState` semaphore through it, so every `_DispatchState` construction site (`run_loop`, plus the `run_tick`/`_dispatch_one` fallbacks) is governed by one helper. `run_loop` also clamps `slots_available` to `_effective_run_cap` so it never spawns a `_dispatch_one` task that would park on a size-1 semaphore. Because the semaphore wraps the **entire** Run including candidate reservation, size 1 guarantees ≤1 in-flight Run and automatically covers resumed/wake/operator-reply re-dispatch — all funnel through `_dispatch_one`. The originally-proposed gate in `_reserve_candidate`/`_reserve_specific_candidate` on `binding.is_remote` was considered and **rejected as redundant** under the semaphore (it would add a branch to two hot reservation functions for the same guarantee). Chosen over building over-SSH remote worktrees (the deferred v2), which buys little throughput — fleet concurrency is *across* bindings, not within one. Local bindings are unchanged (still `config.run_cap`).
- **Secrets/env are the remote host's job (contract reaffirmed).** `_remote_exports` forwards only `SYMPHONY_ISSUE_ID`/`TERM`/`NO_COLOR` (plus the Plane callback env for Plane bindings). Remote agents draw all secrets/config from the remote host's own env (SSH user profile / repo `.env`), consistent with the sudo/privilege line above and ADR-0011. Symphony forwards nothing sensitive; this holds as the fleet grows.
- **SSH keepalive (implemented 2026-06-19).** `ssh_support.ssh_base_args` now emits `-o ServerAliveInterval=15 -o ServerAliveCountMax=4` so an idle NAT/Tailscale timeout no longer drops the long-lived pi RPC channel (~60s to declare the link dead). Cheap idle-drop protection; it does not survive a true network partition — that needs resume-over-SSH (below).
- **Native session resume cannot engage for remote (v2 follow-up).** The resume eligibility check (`session_continuity.py`) does local-filesystem `.exists()`/`glob()` against `~/.pi/agent/sessions/`, but a remote pi run's transcript lives on the *remote* host; `_dispatch_cwd` even hands it the local `homelab_repo_path`. So `evaluate_resume_eligibility` always returns `session-absent` → refeed → cold re-dispatch. True recovery needs reading/resuming the remote transcript over SSH (sibling of the deferred remote Session Tail). Tracked, not built — keepalive cuts the practical drop risk.
- **Still deferred (v2):** remote orphan reaping (local reapers are `/proc`/`/tmp`-bound), a pre-dispatch reachability gate (a down host fails the Run; startup probe logs reachability non-fatally), remote worktrees, remote Session Tail over SSH, styled host chip.

## Amendment 2026-06-23 — v2: Claude on remote bindings (accepted, Issue #96)

**Decision (accepted):** Lift the v1 "remote ⇒ pi-only" invariant so a `remote:` binding may dispatch the Claude agent. Operator driver: Claude models cannot run under pi, so a remote host that specifically needs Claude has no path today. This amends — does not discard — the centralized-brain / SSH-exec / no-remote-daemon stance: the engine still runs only on aidev; the only thing added on the remote is that `claude` + `tmux` already live there (the original `n8n` target already has both, per Context above).

**Why this is a real build, not a config flip.** pi's remote path is thin because `pi --mode rpc` is a single stdin/stdout pipe SSH satisfies directly. Claude has no headless pipe — ADR-0001 rejected `claude -p`, so `claude_runner.py` drives an **interactive tmux TUI** and is coupled to the *local* host at four seams:
- **tmux transport** — every call is `_tmux(run_func, socket_path, *args)` → `run_func(["tmux","-S",socket,*args], …)` (`claude_runner.py:1352`). `run_func` is already injectable.
- **file I/O** — `load-buffer` reads a *local* `prompt_file` (`:1407`); the completion gate polls *local* `done_file`/`result_file` (`:990-1026`) whose paths are baked into the prompt by `_wrap_prompt` (`:1301`); the idle gate reads the *local* transcript mtime (`:1546`).
- **temp dir** — `mkdtemp` creates the prompt/result/done dir on the *local* FS (`:762-766`).
- **liveness/reaping** — the orphan reaper and pidfile guard read *local* `/proc`/`/tmp` (`:167-308`).

**Chosen approach — a `ClaudeHost` seam, mirroring `RepoHost` (ADR-0012's own pattern).** Introduce `LocalClaudeHost` / `SshClaudeHost` (sibling to `repo_host.py`) exposing exactly the four seams above: `tmux(*args)`, `write_file(path, text)`, `read_file(path)` / `exists(path)`, `make_temp_dir()`. `LocalClaudeHost` is today's behavior verbatim (zero change for local bindings). `SshClaudeHost` wraps tmux argv with `ssh_support.ssh_base_args(remote)` (so the tmux server runs **on the remote**, against a remote-side socket) and routes file ops over the **same SSH ControlMaster** connection rather than a fresh `ssh` per poll (the per-second `capture-pane`/`exists` loop makes connection reuse load-bearing, not optional). `run_claude_agent` takes a `host: ClaudeHost` (default `LocalClaudeHost`); `RoutingAgentAdapter` builds an `SshClaudeHost` for `binding.is_remote` + Claude. The reverse-tunnel tracker callback is unchanged — it is already wired for remote pi.

**Rejected:** proxying the *local* tmux server over SSH per command (chatty, fragile, and the prompt/result/done files would still be on the wrong host); and a remote Claude daemon (re-rejected for the same reasons ADR-0012 rejected `symphony-remote`).

**Minimal first slice (what to build; everything else stays deferred).** Non-persist remote Claude only — a single one-shot turn:
1. `config`: allow `default_agent: claude` (or per-issue Claude model) on a `remote:` binding; keep `type: coding` + the per-binding serialize-to-1 semaphore (already in place — guarantees ≤1 tmux session per remote tree, so no warm/persist race to reason about).
2. `ClaudeHost` seam + `SshClaudeHost` with a pooled SSH ControlMaster.
3. `RoutingAgentAdapter`: route remote+Claude to the seam-backed runner instead of raising (`agent_runner.py:1121-1131`).
4. Liveness/reaping: lean on SSH process-group kill on channel close (as remote pi does) + a boot sweep that `ssh host tmux kill-server` on stale remote sockets; **no** remote `/proc` guard in v1 of this slice.

**Explicitly deferred (inherit remote-pi's existing gaps):** `claude_persist` warm sessions + send-keys live steering over SSH (remote Claude is park-and-reply, matching non-persist Claude), remote Session Tail, remote native session resume (cold re-dispatch, same as remote pi), remote orphan reaping beyond boot-sweep.

**Build + verification posture.** The TUI timings and detectors in `claude_runner.py` (`PASTE_SETTLE_SECONDS`, `_ready_pattern_seen`, the paste/Enter race retry, the modal regexes `_MODAL_*`/`_QUESTION_*`) were all hardware-calibrated against a *local* terminal. Driving the same TUI through an SSH-mediated tmux changes latency and redraw behavior, so this slice **must be calibrated live against the `n8n` host** (the existing remote target) before it can be trusted unattended — it cannot be validated by unit tests alone. Sequence: A) `ClaudeHost` seam + local refactor (behavior-identical, unit-tested), B) `SshClaudeHost` + config/routing, C) live `n8n` smoke + timing calibration, then flip status to `accepted`.
