- (technique) started Job to Be Done
- (idea) JTBD-1: When triager opens NewIssueModal, let them fire create faster than alt-tabbing to the blue button — keyboard is the muscle, not the mouse.
- (idea) JTBD-2: When support engineer is mid-reply and run state changes, give them a key to re-evaluate send-vs-comment without reading the chrome.
- (idea) JTBD-3: When power user is in flow, make Enter semantics predictable — Enter=newline, Cmd/Ctrl+Enter=commit, so they never pause to check.
- (idea) JTBD-4: When screen-reader user opens modal, announce submit key, current guard state, and whether attachment upload is blocking submit.
- (idea) JTBD-5: When mobile user has no Cmd key, fall back to a button-only path so shortcut absence never blocks them.
- (idea) JTBD-6: When IME composition is open, never swallow the candidate-confirming Enter — commit only on the *completed* composition Enter.
- (idea) JTBD-7: When novice hits Enter expecting send and gets a newline, give a one-shot affordance so they discover the real submit key in under 30s.
- (idea) JTBD-8: When duplicate-fire risk exists, make a second Cmd+Enter a no-op visibly (button disabled, click ignored) so people learn the submit window.
- (idea) JTBD-9: When staged controls are mid-render, defer submit enable until both stage flags resolve — no flash of ghost-enabled button.
- (idea) JTBD-10: When QA writes an automated test, drive the modal by shortcut and assert isPending disables subsequent submits — no live API calls.
- (idea) JTBD-11: When localization changes button label, do not bind shortcut to label text — bind to behavior so translation never breaks the key.
- (idea) JTBD-12: When ops watches rollout, attribute shortcut-driven submits vs button submits separately so analytics can prove usage before removing the button.
- (idea) JTBD-13: When conflict with browser shortcut exists (Cmd+W, Cmd+R), explicitly *not* override — preserve browser trust except where user opts in.
- (idea) JTBD-14: When attachment upload is mid-flight, make the submit shortcut show progress instead of silently failing — guard visibly not just at state level.
- (idea) JTBD-15: When slash picker is open, never let Cmd+Enter also pick the highlighted option — shortcut must defer to picker focus.
- (idea) JTBD-16: When power-user opens many modals in sequence, position focus on the textarea so first keystroke is the message, not the title.
- (idea) JTBD-17: When tests run in CI, run the shortcut path and the button path against the same component to catch divergence between keydown and click handlers.
- (idea) JTBD-18: When user customizes shortcuts, persist per-user, not per-org — collision space is personal.
- (idea) JTBD-19: When novice needs help, show the shortcut inline once and let them dismiss — never teach by modal popup.
- (idea) JTBD-20: When agent-run automation scripts Podium, skip the shortcut layer entirely and call the submit handler — keep API surface stable.
- (technique) started Constraint Mapping
- (idea) CM-1: Hard constraint — textarea must still accept Enter=newline when shortcut does not match. Therefore keydown must early-return on plain Enter without modifier.
- (idea) CM-2: Hard constraint — isComposing / keyCode 229 must not submit. This blocks CJK, Vietnamese, and other IME users from accidental submit on Enter.
- (idea) CM-3: Hard constraint — if slash picker is open, Enter selects picker item; shortcut cannot fire. Picker wins on Enter, period.
- (idea) CM-4: Hard constraint — same guard list as button: blank description blocked, isPending blocked, attachment upload blocked. No shortcut-only bypass.
- (idea) CM-5: Hard constraint — slash picker combobox item selected by Enter must not also trigger submit. This means picker Enter must stopPropagation before shortcut reads.
- (idea) CM-6: Hard constraint — Escape closes picker first, then modal/flyout. So Escape is already layered; shortcut handler must not also close on Escape.
- (idea) CM-7: Platform constraint — Cmd on macOS, Ctrl on Windows/Linux. Use a normalized 'primary modifier' resolver; do not bind two separate handlers.
- (idea) CM-8: Platform constraint — Cmd+Enter on macOS already used by Slack, GitHub, Linear for 'send message.' Convention aligns us with the user's mental model.
- (idea) CM-9: Browser constraint — inside a textarea, browser-level shortcuts (Cmd+R, Cmd+W) do not fire by default. Safe to use Cmd+Enter here.
- (idea) CM-10: Screen-reader constraint — keyboard shortcut must be announced in modal's accessible description so AT users know submit key exists.
- (idea) CM-11: Focus constraint — shortcut must only fire when focus is *inside* the composer textarea. Outer-page Cmd+Enter must not silently open submit.
- (idea) CM-12: Mobile constraint — no Cmd key. Mobile users use button or, possibly, swipe/Enter on external keyboard. Shortcut is irrelevant; button must remain.
- (idea) CM-13: Key-repeat constraint — holding Cmd+Enter must NOT auto-fire submits. Suppress repeated keydown events (event.repeat === true) for shortcut.
- (idea) CM-14: Staged-controls constraint — submit button is disabled until stage flags resolve. Shortcut must mirror the disabled state — no submit while disabled.
- (idea) CM-15: React constraint — handlers re-create on each render. The keydown listener must use a stable ref or be re-attached on deps change.
- (idea) CM-16: Test constraint — must work in jsdom which simulates isComposing=false. Tests must explicitly set composition state to verify shortcut defers.
- (idea) CM-17: i18n constraint — do not bind shortcut to button label text. Bind to aria-keyshortcuts attribute and let localization update labels only.
- (idea) CM-18: Existing-architecture constraint — SlashPickerTextarea is shared. Adding shortcut to one composer should not duplicate logic; lift to SlashPickerTextarea?
- (idea) CM-19: Form-submission constraint — textarea inside a form means plain Enter is captured by form submit by default in some browsers. Verify form has no implicit submit.
- (idea) CM-20: Soft constraint — discoverability. User must learn shortcut exists. Inline hint near submit button (e.g. '⏎ Create' label) is the cheapest discoverability lever.
- (technique) started Role Playing
- (idea) RP-1 macOS user: Cmd+Enter is muscle memory from Messages/Slack/GitHub. Aligning with that earns trust immediately; Ctrl+Enter on macOS would feel broken.
- (idea) RP-2 Windows/Linux user: Ctrl+Enter is the equivalent convention from Outlook/Teams/GitHub. Use Ctrl on non-mac to match — don't ship Cmd-everywhere.
- (idea) RP-3 Keyboard-only user: Tab-traversal must reach textarea first (already true for modal). Shortcut is the natural completion of their existing flow.
- (idea) RP-4 Screen-reader user (NVDA/JAWS): aria-keyshortcuts='Control+Enter' on textarea announces the binding. Without it, shortcut is invisible to AT users.
- (idea) RP-5 Voice-control user (Dragon): cannot speak 'press Command Enter' reliably. They need a labeled button or numeric mnemonics — shortcut is no help here.
- (idea) RP-6 IME user (Japanese): Enter during candidate selection must NOT submit. isComposing gate is non-negotiable; this is the single most-cited failure mode.
- (idea) RP-7 IME user (Chinese): same — composition state must be checked on every keydown. CJK users will report this within hours of release if broken.
- (idea) RP-8 Power user with muscle memory: wants shortcut identical across NewIssueModal, ReplyComposer, and any future composer. ONE handler, shared utility.
- (idea) RP-9 Novice user: hits Enter, expects send, gets newline. Need a one-time inline hint 'Press ⌘+Enter to send' that fades after first submit.
- (idea) RP-10 Test/QA engineer: wants to assert that keydown event with isComposing=true is ignored. Test must simulate composition, not just keypress.
- (idea) RP-11 Ops/on-call engineer: wants metric — shortcut-driven submits per hour. Distinguishes real muscle-memory adoption from accidental Enter spam.
- (idea) RP-12 Localization manager (German locale): button label changes to 'Erstellen'. Shortcut binding must not change. aria-keyshortcuts survives translation.
- (idea) RP-13 Mobile user (iPad with Magic Keyboard): Cmd+Enter works in Magic Keyboard but not on touchscreen. Button must remain primary on touch.
- (idea) RP-14 Touch-only mobile: no shortcut possible. Button is the only path. Verify button still works in touch-only mode — never shortcut-only UI.
- (idea) RP-15 Power user with custom keymap (Karabiner): user may remap Cmd. Don't fight remaps — bind to the resolved metaKey/ctrlKey, not the key letter.
- (idea) RP-16 Operator in incident: opens flyout reply, types fast, hits Cmd+Enter, expects comment posted as /comment not /reply because run is closed. Verify guard.
- (idea) RP-17 Triager filing 30 issues per hour: Cmd+Enter saves ~1s per issue. Real productivity gain — this is the JTBD primary persona.
- (idea) RP-18 New engineer onboarding: discovers shortcut via inline hint, uses it, never reads docs. Hint is the only doc that matters for novice.
- (idea) RP-19 A11y auditor: will check tab order, aria-keyshortcuts, role=button vs role=textbox, focus management on open/close. Ship shortcut WITH audit-ready attrs.
- (idea) RP-20 Sysadmin customizing app for org: needs no shortcut customization at MVP. Org-level config is YAGNI; per-user config is even more so.
- (technique) started Morphological Analysis
- (idea) MA-1 Trigger axis — Enter alone vs Cmd+Enter vs Ctrl+Enter vs Shift+Enter. Choose Cmd/Ctrl+Enter; reject plain Enter (newline) and Shift+Enter (newline convention).
- (idea) MA-2 Modifier axis — Cmd on macOS, Ctrl on Windows/Linux, none on mobile. Single 'isPrimaryModifier' check; do not branch in render.
- (idea) MA-3 Scope axis — global page vs component-scoped (modal/flyout). Component-scoped wins — global is a power-user feature, not MVP.
- (idea) MA-4 Focus axis — focus-bound (only when textarea focused) vs always-on. Focus-bound is the only safe default; global risks accidental submits.
- (idea) MA-5 State axis — enabled always vs enabled only when guards pass. Shortcut must check the *exact* same guards as button on every keydown.
- (idea) MA-6 Feedback axis — silent submit vs brief button highlight vs toast vs animation. Brief button highlight (200ms pulse) is the cheapest discoverable feedback.
- (idea) MA-7 Discoverability axis — hidden vs inline hint vs modal popup vs onboarding tour. Inline hint on button ('⏎ ⌘+Enter') wins; others are noise.
- (idea) MA-8 Configuration axis — hard-coded vs user-customizable vs org-customizable. Hard-coded for MVP; customization is YAGNI.
- (idea) MA-9 Conflict axis — block on conflict vs warn on conflict vs no-op. Block by checking modifiers; do not warn (warnings are noise).
- (idea) MA-10 Picker-axis — ignore picker entirely vs defer to picker focus vs share picker state. Defer to picker focus — picker Enter takes priority.
- (idea) MA-11 IME-axis — ignore isComposing vs handle keyCode 229 vs block on Enter only. Block Enter when isComposing or keyCode 229; pass everything else.
- (idea) MA-12 Key-repeat axis — fire on each repeat vs block repeat vs block only when guarded. Block repeat entirely; submit is idempotent at human scale.
- (idea) MA-13 Test axis — manual QA only vs jsdom unit vs Playwright integration vs visual regression. jsdom unit for handler; Playwright for end-to-end with composition simulation.
- (idea) MA-14 Telemetry axis — fire-and-forget vs anonymous event vs opt-in vs opt-out. Anonymous event with disabled-by-default flag; opt-in by org.
- (idea) MA-15 Failure axis — silent fail vs toast vs focus return to textarea vs shake animation. Focus return to textarea + silent retry; toast is excessive.
- (idea) MA-16 Attachment axis — block submit while uploading vs allow text-only submit vs queue submit. Block; the existing handler already blocks.
- (idea) MA-17 Staged-controls axis — submit disabled until staged; shortcut must mirror. Use the same disabled prop the button reads from.
- (idea) MA-18 Mobile axis — hide shortcut hint vs show as text vs show as icon. Show as text-only on mobile (no key icon, misleading); show on desktop.
- (idea) MA-19 Localization axis — translate hint text vs keep hint untranslated vs use platform key glyph. Translate the verb ('Send'); keep ⌘/Ctrl glyph literal.
- (idea) MA-20 Architecture axis — duplicate handler per composer vs lift to SlashPickerTextarea vs custom hook. Custom hook 'useComposerShortcut' called by both composers is the cleanest seam.
- (technique) started Reverse Brainstorming + Cursed Genie
- (idea) RB-1 Failure: shortcut submits while slash picker is open, hijacking Enter from the combobox. Fix — picker focus owns Enter; shortcut checks picker state.
- (idea) RB-2 Failure: shortcut submits blank description because guard list was duplicated and one path skipped 'description.trim() !== '''. Fix — single source of truth for guards.
- (idea) RB-3 Failure: IME user on Japanese keyboard hits Enter to confirm candidate; app submits half-formed message. Fix — isComposing and keyCode 229 gate.
- (idea) RB-4 Failure: Cmd+Enter inside title input (not description) silently creates an issue because handler attached to wrong element. Fix — attach to textarea only, scope by element type.
- (idea) RB-5 Failure: holding Cmd+Enter floods server with duplicate creates because key-repeat fires the handler. Fix — event.repeat === true → ignore.
- (idea) RB-6 Failure: shortcut fires while attachment upload is mid-flight, blocking submit silently; user clicks button and nothing happens. Fix — explicit 'uploading' state blocks shortcut AND button visibly.
- (idea) RB-7 Failure: screen-reader announces 'button Create' but never mentions shortcut exists. Fix — aria-keyshortcuts='Control+Enter' on textarea or submit button.
- (idea) RB-8 Failure: macOS Cmd+W closes tab while user expected shortcut to send; app loses draft. Fix — do not override browser chrome shortcuts; shortcut is in-textarea only.
- (idea) RB-9 Failure: Windows Ctrl+Enter on Firefox triggers form submit before our handler runs, double-firing. Fix — preventDefault on the form submit when in textarea.
- (idea) RB-10 Failure: shortcut works in modal but ReplyComposer uses /reply vs /comment logic; Cmd+Enter posts as /reply even when run is closed and comment was correct. Fix — defer to handler's guard, which already checks run state.
- (idea) RB-11 Failure: shortcut bypasses isPending state because button uses disabled but keydown reads a stale closure. Fix — ref-guard or read state at keydown time, not render time.
- (idea) RB-12 Cursed genie: you must add a 5-key chord shortcut. User reports RSI within a week. Reject chord; single Cmd+Enter only.
- (idea) RB-13 Cursed genie: shortcut must work even when textarea is *not* focused. Now any global keystroke can submit; chaos. Reject; focus-bound always.
- (idea) RB-14 Cursed genie: shortcut must work on mobile. Now we're inventing on-screen key combos on touch. Reject; button is the touch path.
- (idea) RB-15 Cursed genie: shortcut must be discoverable without any UI hint. Now we're shipping hidden affordances, losing novices. Reject; inline hint mandatory.
- (idea) RB-16 Cursed genie: every shortcut must trigger a confirmation modal to prevent accidents. Now no one uses the shortcut. Reject; guards already prevent.
- (idea) RB-17 Cursed genie: shortcut must be the same across all browsers AND override Ctrl+W in Firefox. Reject; never override browser chrome.
- (idea) RB-18 Cursed genie: every composer gets a unique shortcut. Now users must memorize N bindings. Reject; one shortcut, one mental model.
- (idea) RB-19 Cursed genie: shortcut must work even when JS is disabled. Reject; this is a JS-required SPA, not a static form.
- (idea) RB-20 Inverse success criterion — the best shortcut is the one users never think about. If we ship Cmd+Enter and three weeks later no one mentions it, we won.

- (decision) Convergence: ship useComposerSubmitShortcut hook (Cmd/Ctrl+Enter, focus-bound, IME-aware, picker-deferring, repeat-blocking, guard-mirroring). Wire into NewIssueModal and ReplyComposer. Inline hint on submit button. aria-keyshortcuts attr. One test file, four states. No palette, no customization, no analytics in v1. Defer chord, voice, org overrides.
- (insight) Primary recommendation: ship only Cmd+Enter and Ctrl+Enter as additive submit shortcuts in NewIssueModal and the parked-issue ReplyComposer. Plain Enter remains newline; the visible Create/Send buttons remain the universal touch, voice, and fallback path.
- (insight) Source-grounded seam: SlashPickerTextarea currently owns slash-picker open state and consumes Enter for picker selection, so avoid a global listener or a new shortcut manager/hook. Add one optional submit-shortcut callback to the existing shared textarea; it can suppress the chord while its picker is open, during IME composition, or on key repeat, then each caller invokes its existing submit path and guards.
- (insight) Conflict posture: no browser-wide shortcut interception. React only to exact Meta-or-Control plus Enter while the relevant textarea has focus, call preventDefault only for that chord, and advertise both with a compact visible hint and aria-keyshortcuts. This minimizes browser/OS/assistive-tech collision risk without claiming conflicts are impossible.
- (insight) Correctness contract: keyboard and button converge on the same create/reply action, preserving blank, pending, upload, run-state, staged-control, and comment-vs-reply checks. The disabled button state remains the source of user feedback; no toast, animation, telemetry, or confirmation dialog is needed.
- (insight) Verification should extend existing Playwright specs: both modifiers submit, plain and Shift+Enter preserve multiline text, open slash picker and active IME do not submit, disabled/pending/upload states do not submit, and repeated keydown cannot create duplicate requests.
- (decision) Converged v1: modify the existing shared SlashPickerTextarea plus the two existing composers and their existing Playwright specs; no new file, global shortcut registry, OS-sniffing layer, command palette, customization, analytics, mobile gesture, or broader keyboard-navigation scope.
