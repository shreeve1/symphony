## Core Claim
Claude Code now ships **built-in git worktree support** (`claude --worktree`),
letting agents run in parallel in isolated worktrees without interfering with one
another — each agent gets its own worktree and works independently, and the tool
"absorbs the complexity" of git worktrees by managing each worktree's lifecycle
tied to a single agent's work.

## Mechanics
The video walks through two paths side by side:

**Plain git worktrees (manual baseline):**
- From the main repo (his "course video manager") he runs `git status` to show he's on `main`.
- He runs `git worktree add UI updates`, which creates a new folder `UI updates`.
- `cd` into it + `git status` shows it's on a branch called `UI updates` — a separate git state / branch from main, living in its own folder.
- `touch foobar.md` creates a file. VS Code's source-control panel now shows *both* the main repo and the worktree as separate sources; he stages and commits to the `UI updates` branch, publishes the branch, and opens a PR back to main — all inside the IDE.
- After merging the PR, he `cd`s back to the main directory, runs `git pull`, and `foobar.md` appears on `main`.
- He removes the worktree with either `rm -rf UI updates` or `git worktree remove UI updates` ("fairly sure that does kind of the same thing"). Source control then shows just the single main worktree again.

**Claude's version (`claude --worktree`):**
- Running `claude --worktree` creates a worktree at `.claude/worktrees/<random-name>` (e.g. "cheerful coalescing worth", "delightful dazzling sketch") — auto-generated whimsical branch/worktree names. "It's just using git worktree under the hood."
- He tells Claude to delete `foobar.md`; Claude searches for the file, removes it, runs `git status` / `git log`, stages, commits, and pushes, then opens a PR. "Claude is really just doing the same setup as we had before. It's just automatically naming you a branch."
- On `Ctrl+C` (quit), Claude offers to **keep or remove the worktree**. Removing an un-pushed worktree loses changes/commit history; he pushes first instead, then removes.

## Architecture
- Built on top of plain `git worktree` — "it's just using git worktree under the hood."
- Worktrees are created under `.claude/worktrees/<auto-generated-name>/`.
- Branch/worktree names are auto-generated (random multi-word names).
- IDE integration (VS Code, "probably cursor too") surfaces each worktree as its own entry in the source-control panel for independent staging/commit/PR.
- Lifecycle (create → work → keep/remove) is managed by Claude and tied to a single agent's session.
- **Sub-agents support worktrees too** (per the announcement he references): you can spin up a sub-agent, have it make a PR back to main, and orchestrate from above — but this is **opt-in**; Claude won't use worktrees for its agents automatically.

## The Paper Cut (creator-reported)
His main caveat, discovered live: the commit didn't land where he expected.
- After the Claude worktree run, `git status` showed `main` and `git log` showed the delete commit *in main's history*.
- In a fresh session he asked Claude to add `foobar.md`, commit, and push to the branch. Claude was on worktree branch "delightful dazzling sketch" but reported "up to date with origin main" — "the source of the branch ... is main even though we're on a different worktree."
- A **safety hook blocked the push** and printed a command; he manually ran `git push origin worktree-delightful-dazzling-sketch`, which created the correctly-named remote branch.
- Takeaway in his words: "If you're using claude worktrees, your agent needs to push with a specific named branch. Otherwise, by default, it's going to be the one that it created the worktree from — in this case main." Risk: "if your main branch is not protected and you create a worktree from it, you might end up accidentally pushing commits to main when you wanted to push them to a separate branch." He notes this is promptable and "I don't hate it."

## When It Applies
- For: parallelizing Claude Code work — "every single time you have an idea or you want to do something you can just instantly spin up a new worktree, get things working, and ping that back to main as a PR." Makes parallelization "a lot more free than it was before." Also for sub-agent orchestration (spin up sub-agent → PR back to main).
- Caution: requires explicit named-branch pushing to avoid accidental commits to main; sub-agent worktree use is opt-in, not automatic; if main isn't branch-protected, accidental pushes to main are possible.

## Builder Takeaway
A developer should walk away wanting to default to `claude --worktree` for most
sessions to get cheap parallelization, while (a) protecting `main`, (b) prompting
the agent to push to a specific named branch, and (c) deciding deliberately
whether to keep or remove each worktree on exit so un-pushed work isn't lost.
Worktrees + sub-agents enable an orchestrated "fan out, PR back to main" workflow.
